## 0. 设计原则与对齐声明

### 0.1 数据采集来源＝「混合派生模型」（服务端权威子集 + 客户端上报经状态哈希校验）
> **v1.1 修订**：原 v1.0「服务端从权威仿真逐帧派生」与已定架构冲突——主架构 §5.3 服务器是**中继 + 事件注入、不跑模拟**，Entity 层无仿真；服务器为 .NET 8 + Fantasy-Net，**跑不了客户端 Unity DOTS/Burst 仿真**，无法逐帧派生依赖位置/碰撞/CA 像素的事件。协议 §6.1 亦明确：`Rank/Kills/Deaths/MaterialContribution` 是**确定性模拟量，客户端在 `MatchEnded` 时从本地 sim 派生**，`G2C_MatchSettle` 仅补服务端权威的 `MmrDelta`。故改为下列**混合派生模型（MVP 可行，对应服务端评审选项 A）**。

- 本作是**帧同步确定性仿真**（30Hz 逻辑 / 60Hz 渲染，GGPO 回滚，每 60 帧状态哈希比对）。但**服务端不持有仿真**，因此事件按「能否仅由输入流 + 场景时间线 + 镜像/冷却追踪器推导」分两类（详见 §2.1）：
  - **① 服务端权威子集**（无需仿真）：E1 `match_start`、E2 `spell_cast`、E6 `item_pickup`、E7 `spell_switch`、E8 `dash_used`、场景 spawn/despawn —— 由 `C2G_Input` 输入流（ActionFlags 边沿 + 镜像 `SelectedSpellSlot` + 权威 `SpellIds` + 冷却时间线）、`SceneScheduler` 服务端注入时间线、服务端状态机推导。
  - **② 客户端上报 + 状态哈希校验**（依赖位置/碰撞/CA 像素）：E3 `spell_hit`、E4 `knockback_applied`、E5 `boundary_kill`、`MaterialContribution` —— 客户端带外上报（**不进 `InputPayload`，不污染哈希**），服务端用**既有的每 60 帧状态哈希多数派裁决（协议 §5.4 / §9.2）校验**，死亡/状态变化须与多数派哈希一致才采纳，分歧则告警/拒。即「verify-don't-trust」，保留抗作弊精神且 MVP 可落地。
- **红线（不变）**：**所有**分析事件**绝不**进入 `InputPayload`，**绝不**参与状态哈希（对齐协议 §3 / §8 确定性红线；`PauseMenu`/`QuickChat` 不进 `InputPayload` 同理）。
- 与 `结算界面规格.md` §3.3 一致：**「数据来自权威」**——客户端只展示、不重算结算；E3/E4/E5 类事件的**客户端上报值须经状态哈希多数派校验**后，服务端才在 `G2C_MatchSettle` 汇编中采纳（§2.4 校验门）。
- 唯一一次性下行结算包 `G2C_MatchSettle`（protobuf 低频）即权威聚合来源，本规格的 `match_settle` 事件与其 **1:1 字段对齐**（见 §2.4 / 附录映射表）。`G2C_MatchSettle` 由服务端**汇编 + 中转 + 补 `MmrDelta`**（非服务端仿真源头）。

### 0.2 时间轴＝逻辑帧 `Frame`（30Hz），非墙钟
- 所有对局内事件以**逻辑帧号 `Frame`**（uint64，30Hz）为时间戳：`t_sec = Frame / 30`。
- 好处：① 与回放/哈希对齐，可事后精确重建；② 跨端一致，无 NTP 抖动；③ 客户端 UI 耗时（帧率/卡顿）才用墙钟，且走独立带外通道（非仿真）。

### 0.3 对齐 ADR D4「无 HP」——埋点零 HP 字段
- **唯一死因＝出界**（百战天虫规则，ADR D1/D4）。因此：
  - 不存在 `hp_death` / `damage_taken` / `health` 类事件或字段；
  - 死因只有一种：`boundary_out`（出界），由 `boundary_kill`（出界即死）事件承载；
  - 结算项**无 HP 项**，与 `G2C_MatchSettle` 完全一致（`Deaths ∈ {0,1}`，胜者=0，其余=1）。
- 环境危害（火/毒/酸/熔岩）在 MVP 主职=**逼走位/控场**，**非致死**（GDD Q1 / ADR §4 / 数值 §3.2）——分析侧不统计"环境致死"，只统计"环境材料改造量"与"环境触发的击退冲量（作为击杀归因的 proximate push）"。

### 0.4 与 `G2C_MatchSettle` 字段对齐表（权威基准）

| `G2C_MatchSettle` / `SettlePlayer` 字段 | 类型 | 是否确定性 | 分析侧落点 |
|---|---|---|---|
| `MatchId` | uint32 | 是 | `match_settle.match_id` |
| `TotalFrames` | uint32 | 是 | `match_settle.total_frames` → 推导 `match_duration_ms` |
| `SettlePlayer.PlayerId` | uint32 | 是 | 玩家主键 |
| `SettlePlayer.Rank` | uint32 | 是 | 名次（1=冠军） |
| `SettlePlayer.Kills` | uint32 | 是 | 击杀数 K（按 §3 归因） |
| `SettlePlayer.Deaths` | uint32 | 是 | 死亡数 D（0 或 1） |
| `SettlePlayer.MaterialContribution` | uint32 | 是 | 材料改造贡献像素数 |
| `SettlePlayer.MmrDelta` | int32 | **否（服务端权威，ELO 变体）** | 排位 MMR 变化 |
| `YourselfRank` | uint32 | 是 | 本地玩家名次（可派生） |

> 注：`G2C_MatchSettle` 客户端本地可从 sim 推导除 `MmrDelta` 外所有字段（协议 §6.1）；分析侧**服务端落盘**全量，不依赖客户端上报。

## 1. 核心指标字典（Metric Dictionary）

> 口径约定：所有比率分母明确；"场均"指**每局每玩家**；维度至少含 `mode`（Quick/Custom/Ranked）、`boundary_mode`（Kill/Bounce/Wrap）、`player_count`、`date`。

### 1.1 留存：次日留存 D1 / 7 日留存 D7
| 项 | 定义 |
|---|---|
| **口径** | 以玩家**首次进行对局的日期**为第 0 日（cohort 日）。`Dn 留存` = 在第 0 日之后第 n 日**至少进行 1 局对局**的玩家数 ÷ 第 0 日新增玩家数。 |
| **计算** | `Dn = count(distinct player_id where has_match_on_day(day0+n)) / count(distinct player_id where first_match_day=day0)` |
| **维度切分** | 按 `mode`、`boundary_mode`、获客渠道（预留）、首局胜负、首局名次分桶。 |
| **目标/告警** | 暂无历史基线；上线首周建基线。告警：D1 < 25% 或 D7 < 8% 触发复盘（休闲竞技参考线）。 |
| **依赖（重要）** | **需要稳定匿名玩家 ID**。GDD §9 明确"v1 暂不实现账号持久化进度"，但留存 cohort 必须有稳定 key。→ 建议：首启时生成**匿名 Install/Device ID**（独立于进度系统、仅用于 cohort），不阻塞 MVP（见 §6 P0-1）。 |

### 1.2 对局时长 & TTK
| 指标 | 定义 / 计算 | 备注 |
|---|---|---|
| **单局时长 MatchDuration** | `total_frames / 30 * 1000` ms（直接来自 `G2C_MatchSettle.TotalFrames`） | GDD §3 目标：4 人局 < 8 min；核心体验"单局 5 分钟内结束"。 |
| **首杀时间 TTK-FirstKill** | 从对局首帧到**第一例** `boundary_kill` 的 `Frame` 差 ÷ 30 | 数值 §13 试玩验收指标"首杀时间"；过长=节奏拖沓。 |
| **平均淘汰间隔 TTK-ElimGap** | 相邻两次 `boundary_kill` 的帧间隔均值 | 反映中后期节奏密度。 |
| **时长分布分位** | P10 / P50 / P90 / P99 单局时长 | 监控长尾（>8min 占比），避免僵局。 |

> 命名说明：任务书称"对局时长（TTK）"；本规格将 **TTK 细分为 MatchDuration（总时长）与 TTK-FirstKill（首杀时间）** 两个可行动 KPI，避免混淆。

### 1.3 场均击杀 / 出界数
| 指标 | 定义 / 计算 | 分析注意 |
|---|---|---|
| **场均击杀 AvgKills** | `sum(Kills) / match_count`（每局每玩家均值） | 纯 BR 中总击杀 ≈ 淘汰数 = `N − 存活数`，故 AvgKills ≈ `(N−存活)/N`，**绝对值信息量低**；价值在**击杀归属分布**与**自灭率**（见 §3.4）。 |
| **场均出界数 AvgElim** | `count(boundary_kill) / match_count` | 与 AvgKills 在归因正确时应≈相等（每次出界对应 1 次击杀归属，或自灭）。 |
| **击杀信用分布 KillCreditShare** | 各玩家 `Kills` 占全局比；分段（0 杀 / 1 杀 / 2+ 杀占比） | 检验"是否总同一人收割"。 |
| **自灭率 SelfElimRate** | `self_boundary_kill / total_boundary_kill` | **平衡关键**：自灭过高=边界过严/走位惩罚重；过低=推人机制失效。 |

### 1.4 MMR 分布（排位 Ranked 专属）
| 指标 | 定义 / 计算 |
|---|---|
| **MMR 模型** | ELO 变体：`K=32`，`BaseMmr=1000`（GDD §9）。`MmrDelta_i = round(K * (S_i − E_i))`，其中名次分 `S_i = (N − Rank_i)/(N−1) ∈ [0,1]`（冠军=1，末位=0），期望 `E_i = 1/N`（均匀先验，最简「按名次结算」；后端可用对手赛前 MMR 细化 `E_i`）。 |
| **起点** | 新排位玩家 `Mmr=1000`。 |
| **分布看板** | MMR 直方图（分桶，如 800/900/1000/1100/1200…）、中位数、P25/P75/P90、收敛速度（周环比标准差变化）。 |
| **健康度** | 中段（900–1100）应聚集且缓慢扩散；若大量玩家快速贴到两端=匹配分池问题。 |

## 2. 埋点事件方案（Tracking Plan）

### 2.1 事件总表

| # | 事件名 | 触发时机 | 来源 | 主要落点 / 上卷 |
|---|---|---|---|---|
| E1 | `match_start` | 对局状态机进入 `Battle`（首帧） | 服务端权威子集（状态机，天然权威） | 留存/时长分母、漏斗起点 |
| E2 | `spell_cast` | 玩家 `Attack` 位触发且法术实际施放（CD/Mana 通过） | 服务端权威子集（Attack 边沿 + 镜像 `SelectedSpellSlot` + 权威 `SpellIds` + 冷却时间线；`mana_blocked` 若服务端无 mana 状态标"未知"） | 施法分布、双闸健康 |
| E3 | `spell_hit` | 弹体/范围命中**玩家**像素采样成功 | **客户端上报 + 状态哈希校验**（依赖位置/碰撞，服务端无仿真） | 命中率、击退归因输入 |
| E4 | `knockback_applied` | 任意击退冲量 `K` 施加到某玩家（玩家/生物/环境/反应） | **客户端上报 + 状态哈希校验**（依赖位置/碰撞） | 击杀归因（§3）、被推挤出界占比 |
| E5 | `boundary_kill` | 玩家像素中心越过边界 → `PlayerDeath`（出界即死） | **客户端上报 + 状态哈希校验**（依赖位置/CA 像素） | 死因唯一事件、Kills/Deaths 上卷 |
| E6 | `item_pickup` | 玩家拾取地面道具（`Pickup` 位 + 接触） | 服务端权威子集（`Pickup` 边沿；`contested` 用"同帧多 Pickup 边沿"近似） | 道具争夺集中度、Build 分析 |
| E7 | `spell_switch` | `SelectNext`/`SelectPrev` 改变 `SelectedSpellSlot` | 服务端权威子集（SelectNext/Prev 位） | 法术轮切行为 |
| E8 | `dash_used` | `Dash` 位触发且 CD 通过 | 服务端权威子集（`Dash` 边沿 + CD） | 机动性、逃生/突袭分析 |
| E9 | `match_settle` | `MatchEnded` 后下发 `G2C_MatchSettle` | 服务端汇编+中转+补 `MmrDelta`（= wire，1:1 §0.4） | **结算聚合，字段对齐 §0.4** |
| — | `scene_spawn` / `scene_despawn` | 道具/生物刷新与消失 | 服务端权威（SceneScheduler 注入，天然权威） | 刷新节奏、生态分析 |

> **混合派生模型（v1.1，依云澈评审）**：E1/E2/E6/E7/E8 + 场景事件为**服务端权威子集**（仅靠输入流 + 场景时间线 + 镜像/冷却追踪器，无需仿真）；**E3/E4/E5/`MaterialContribution` 依赖位置·碰撞·CA 像素状态，服务端无仿真无法派生，改由客户端带外上报（不进 `InputPayload`），服务端以**每 60 帧状态哈希多数派裁决（协议 §5.4/§9.2）校验**后采纳（verify-don't-trust）。客户端仅另上报独立带外的 `client_perf`（帧率/卡顿/崩溃）与 UI 点击埋点，本文不展开。红线不变：所有事件不进 `InputPayload`、不参与状态哈希。

### 2.2 公共字段（所有 E1–E9 携带）
| 字段 | 类型 | 说明 |
|---|---|---|
| `match_id` | uint32 | = `G2C_MatchSettle.MatchId` |
| `frame` | uint64 | 逻辑帧（30Hz），时间戳轴 |
| `player_id` | uint32 | 主体玩家（E5/E6 中"受害者/拾取者"；E2/E3/E4/E7/E8 中"施动者"） |
| `mode` | enum | Quick / Custom / Ranked |
| `boundary_mode` | enum | Kill / Bounce / Wrap |
| `player_count` | uint8 | 房间人数 N |
| `arena_seed` | uint32 | 确定性地形种子（用于按地图聚合） |
| `is_ranked` | bool | 是否排位（决定是否采 `mmr_delta`） |

### 2.3 各事件字段定义
**E1 `match_start`**：`{公共} + spawn_positions(map player_id→(x,y))`、`start_mmr`（排位各玩家赛前 MMR，用于 E9 校验）。
**E2 `spell_cast`**（服务端权威子集）：`{公共} + spell_id`、`spell_slot`、`element`（火/冰/毒/酸/投掷…）、`cast_result`（ok/cd_blocked；`mana_blocked` 若服务端无 mana 状态则标"未知"，可由客户端补充上报经校验）。`mana_before`/`mana_after` 同理——服务端缺 mana 状态时可标"未知"。
**E3 `spell_hit`**（客户端上报 + 哈希校验）：`{公共(施动者)} + victim_player_id`、`spell_id`、`hit_type`（point/aoe/beam）、`knockback_k`（本次施加击退量，见 K 表）、`contact_frame`（用于归因链）。
**E4 `knockback_applied`**（客户端上报 + 哈希校验）：`{公共(施动者可为玩家/生物/环境)} + victim_player_id`、`source_type`（player/creature/environment/reaction）、`source_id`（若 player 则为 player_id；若 creature 为 entity_id；环境为 material_type）、`knockback_k`、`edge_flag`（受害者是否处于边缘 15–55px，触发出界风险）、`contact_frame`。
**E5 `boundary_kill`**（客户端上报 + 哈希校验）：`{公共(受害者)} + death_frame`、`last_contact_player_id`（§3 归因结果，可空=自灭）、`last_contact_frame`、`proximate_source_type`（出界前最后一推：player/creature/environment/self）、`attribution_window_hit`（bool，是否在归因窗口内）、`survival_frames`（出生到出界）。
**E6 `item_pickup`**：`{公共} + item_type`（武器/卷轴/药水/护盾/速度/炸弹）、`item_id`、`telegraph_lead_frames`（从 Telegraph 预告到落地的帧数，争抢分析）、`contested`（同帧/近帧是否有 ≥2 玩家抢同一道具）。
**E7 `spell_switch`**：`{公共} + from_slot`、`to_slot`、`switch_dir`（next/prev）。
**E8 `dash_used`**：`{公共} + dash_dir`、`dash_distance_px`、`post_dash_pos`、`iframes_active`（MVP=无 i-frames，纯位移，不免疫任何效果）、`knockback_during_dash`（bool，Dash 期间是否被推，验证"Dash≠免推"）。
**E9 `match_settle`**：**严格对齐 `G2C_MatchSettle`**：
```
match_id, total_frames, yourself_rank,
players[]: { player_id, rank, kills, deaths, material_contribution, mmr_delta },
// 派生（分析侧附加，不进 wire）：
mode, boundary_mode, player_count, arena_seed, is_ranked,
match_duration_ms (= total_frames/30*1000),
start_mmr[]（赛前，用于分布）, end_mmr[]（= start+delta）
```

### 2.4 数据质量 & 反作弊
- **红线（不变）**：所有分析事件**绝不**进 `InputPayload`、绝不参与状态哈希（协议 §8）。客户端对局事件不可信——本规格坚守此点；差异仅在"谁算"而非"是否防作弊"。
- **校验门（下发前 sanity gate）**：服务端在**汇编 `G2C_MatchSettle` / 落盘 E1–E9 之前**强制跑：
  1. **字段对齐校验**：`sum(kills) == count(boundary_kill where last_contact_player_id != null)` 且 `sum(deaths) == count(boundary_kill)`（D∈{0,1}）；不一致告警（归因 bug/丢帧）。
  2. **帧连续性**：`match_settle.total_frames` 应 ≈ E5 最大 `death_frame` 的场次；缺口告警（服务器掉帧/日志丢失）。
  3. **客户端上报校验（verify-don't-trust）**：E3/E4/E5/`MaterialContribution` 的客户端上报值，须与**每 60 帧状态哈希多数派裁决（协议 §5.4/§9.2）**一致才采纳；死亡/状态变化与多数派哈希分歧 → 拒收 + 告警（潜在作弊/丢帧）。
- **归因窗口参数化**：`ATTR_WINDOW`（§3）作为可配常量，写入事件便于回溯重算。

## 3. 击杀归因模型（Kill Attribution）：last-contact 玩家归因

### 3.1 铁律（引用 `生物AI规格.md`）
- **生物不直接致死**（生物AI规格 §1：「致死一致（D4）：MVP 生物不直接致死；攻击=击退/控制；死亡只由出界判定」；§7：「MVP 生物不直接致死」）。
- **死亡只由出界判定**（Kill 规则，ADR D1/D4）。
- **击杀归属**（GDD §11 Q3 / ADR §4：「生物/环境致死按『最后接触玩家』归属名次」；生物AI规格 §7：「生物致死按『最后接触玩家』归属名次」）。

→ 推论：**唯一死因是出界；而出界的 proximate push 可能来自玩家/生物/环境/自身，但击杀信用（Kills +1）只授予「最后接触玩家」**。

### 3.2 last-contact 玩家归因算法（客户端确定性计算 + 服务端校验）
> 因服务端无仿真（§0.1），`last_player_contact` 状态由**客户端按确定性规则本地计算**（与 sim 同逻辑、可回滚）；服务端**校验规则一致性 + 状态哈希多数派**（§2.4 校验门）后，在 `G2C_MatchSettle` 汇编中采纳 Kills 信用。算法本身完全确定性，跨端一致。

每个玩家维护状态：
- `last_player_contact_id` / `last_player_contact_frame`：**仅当另一个 PLAYER 对其施加击退/碰撞/接触冲量（`E4.source_type == player`）时更新**。
- 生物/环境/反应（`E4.source_type ∈ {creature, environment, reaction}`）**不覆盖** `last_player_contact_id`（它们是 proximate push，不抢信用，符合 §3.1 铁律）。

出界触发 `E5 boundary_kill` 时：
```
if last_player_contact_id 存在
   AND (death_frame − last_player_contact_frame) <= ATTR_WINDOW:
        credited_player = last_player_contact_id   # Kills+1
else:
        credited_player = NULL                       # 自灭（SelfElim）
```
- **`ATTR_WINDOW`（归因窗口）**：默认 **15s = 450 帧**（可配）。理由：纯自走出界（无近期玩家接触）= 自灭；若窗口过短会把"生物推挤出界但 16s 前被玩家碰过"误判自灭，过长会把远古接触也算信用。窗口值随试玩校准（与数值 §13「首杀时间/单局时长」同批验收）。

### 3.3 边界 case 处理
| Case | 处理 |
|---|---|
| **玩家 A 用重弹把 B 推到边缘，B 自行走出界（无后续接触）** | 仍在 `ATTR_WINDOW` 内 → 信用归 A（last-contact 即 A）。✅ |
| **生物把 B 推出界，B 最近一次玩家接触是 5s 前的 A** | 生物不覆盖 `last_player_contact` → 信用归 A（符合 Q3「生物致死按最后接触玩家」）。✅ |
| **B 全程无玩家接触，自己走/掉出界** | `last_player_contact_id` 空或超窗口 → 自灭（SelfElim），不计入任何人 Kills。✅ |
| **同帧多人出界** | 按 `death_frame` 同帧内以 `player_id` 确定性子排序定名次（与结算 `Rank` 一致）；归因各自独立计算。 |
| **被推挤出界 vs 自灭的判定** | 以 `proximate_source_type` + `last_contact_player_id` 双字段记录，供 §3.4 占比分析。 |

### 3.4 归因校验 / 平衡指标
- **被推挤出界占比 `PushOutRate`** = `count(E5 where proximate_source_type==player) / count(E5)`。数值 §13 风险登记明确监控此值；过高→K 表过强（轻弹乱撸也能推人，违反"仅边缘 15–55px"铁律）。
- **自灭率 `SelfElimRate`** = `count(E5 where credited_player==NULL) / count(E5)`。
- **生物归因占比** = `count(E5 where proximate_source_type==creature and credited_player!=NULL) / count(E5)`：验证"生物致死仍归玩家"的 Q3 落地正确性。
- **助攻 Assists（P1）**：`last_player_contact` 之外、在更短窗口（建议 5s=150 帧）内接触过受害者且非终结者的玩家 → 写入 `PlayerResult.Assists`（对齐 结算界面规格 §3.3 P1 字段）。

- **P0-3｜监控 `PushOutRate` 与被推挤出界占比**：数值 §13 风险。若 > 阈值（试玩定，初版建议单局被推挤出界 > 40%）→ K 表过强，回调节退 K（轻弹 45/重弹 90/爆炸 150）。

- **P1-1｜激活商业化埋桩**：待账号/经济系统落地，启用 §4 事件，接 §5.2 看板。
- **P1-2｜助攻 Assists 与分元素施法/材料改造量**：结算界面规格 §3.3 已留 P1 字段，归因窗口稳定后开启，丰富行为看板。
- **P1-3｜`ATTR_WINDOW` 校准**：随试玩把 15s 默认值调到「自灭率合理、生物归因正确」的区间。

   - **B) 服务端 100% 权威派生**：须把确定性仿真核心移植为服务端可跑的 net8.0 纯 C# 库（headless shadow sim）。大重构，**大概率 post-MVP**，作为未来加固项。

### 7.3 DeviceId / InstallId 格式规范（供端木林落地客户端采集）
> 本规范是埋点稳定 ID 的**单一事实源**；客户端采集逻辑以此为准。

- **生成与持久化**：
  - `DeviceId`：随机 `UUIDv4` 生成**一次**，存**设备级**持久化（如移动端设备作用域存储 / PC 注册表或设备作用域配置），**重装不变** → 主留存 cohort key。
  - `InstallId`：随机 `UUIDv4` 生成**一次**，存**安装级**持久化（应用沙盒），**重装可变** → 区分同设备多开。
- **上报前哈希（不传明文）**：
  - 算法 **SHA-256**；输入＝canonical UUID 字符串（**小写、去连字符**）；输出 **小写 hex、完整 64 字符**（不截断，避免碰撞）。
  - 盐：源为随机 UUID（无 PII），**无需 per-user 盐**；可加**固定 app 命名空间前缀**（静态常量）做跨应用隔离（可选，建议加）。
  - 即：`DeviceId_hash = SHA256( APP_NS + uuid4_device_canonical_lowercase )`，`InstallId_hash` 同理。
- **wire 类型**：`string`（hex）；字段 `C2G_DeviceReport.DeviceId` / `.InstallId`。
- **上报时机**：连接钩子里采集并哈希，**连接 / 重连时一次性**随 `C2G_DeviceReport` 上报；不进 `InputPayload`、不进 30Hz、不 bump ProtocolVersion。
- **服务端（yunche #32）**：Session/PlayerSlot 绑定，供 E1–E9 的 `player_id`/稳定 ID 关联；分析侧 D1/D7 cohort 以 `DeviceId` 为主键，`InstallId` 辅助拆分同设备多开。
- **隐私**：匿名、无明文、不关联账号；`DeviceId` 设备稳定（重装不变）、`InstallId` 安装级（重装可变）。
