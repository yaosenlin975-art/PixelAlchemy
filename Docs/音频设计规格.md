## 0. 一句话定位

> **音频是纯表现层。** 一切"是否发声、发什么声、何时发"都必须由**确定性帧同步模拟事件**驱动；音频引擎内部如何随机化微变体（pitch/volume）无所谓，因为它从不回馈模拟、不参与锁步哈希、不导致 desync。

本档给出：中间件选型、SFX 清单（绑定确定性逻辑事件枚举）、音乐与自适应混音、确定性触发模型、移动端性能预算。全部对齐 ADR 确定性红线。

---

## 1. 概述与目标

| 目标 | 设计意图 |
|------|---------|
| **确定性安全** | 游戏玩法音效 100% 来自锁步模拟事件队列；绝不依赖客户端随机 / 渲染时序 / `Time.deltaTime` |
| **可读性反馈** | 用音色区分法术类别、用边界警示音强化"出界即死"招牌机制（GDD §4.2 / ADR §1） |
| **移动端可承载** | 严格 voice / 内存 / CPU 预算，pixel 游戏音频体量克制 |
| **自适应张力** | 对战 BGM 随战斗强度（存活人数 / 近期交火）动态混音，不硬切 |
| **回滚一致** | 帧回滚（MAX_ROLLBACK=7）时音频事件随之丢弃重建，听完与看完一致 |

## 2. 确定性红线与音频定位（★最关键）

> 摘自 ADR §0/§2 与 UI_UX §1.2 原则"确定性优先"：**任何影响世界状态的操作必须能映射成帧同步输入；元操作（暂停、表情）不进确定性流。** 音频不"影响世界状态"，但其**触发源**必须来自确定性流。

| 维度 | 规则 |
|------|------|
| 音频是否进 `InputPayload` | **否**。音频永不写入 30Hz 输入，永不参与 `C2G_StateHash` / 锁步哈希（对齐 ADR §2 与协议规范 §2 红线） |
| 音频是否进模拟状态快照 | **否**。音频事件不入 `PlayerLayerSnapshot` 状态量（FRAME_SYNC §2.3.3），只入**独立的、可丢弃的**每帧音频事件缓冲 |
| 触发源 | 仅来自确定性模拟在 `ApplyInputsAndStep` / `ReplayForward` 中**同步产生**的 `SimAudioEvent` |
| 客户端随机 | **禁用**于"是否发声 / 发哪种声"的判定；`UnityEngine.Random` / 非种子 `System.Random` / `if (rand < x) Play()` 一律禁止 |
| 渲染时序 | 音频触发不得绑定渲染帧率 / `Time.deltaTime`；绑定逻辑帧号 `Frame` |
| 回滚 | 未确认帧（≤ INPUT_DELAY）的音频事件可被丢弃；确认帧之后的事件提交给音频引擎（见 §4） |

**结论**：音频引擎（Wwise/FMOD）本身"确定性中立"——它既不破坏也不增强锁步。确定性由**我们的集成层**（触发源 + 提交时机）保证，与具体中间件无关。

## 3. 中间件选型：Wwise vs FMOD

### 3.1 对比表

| 维度 | **Wwise (Audiokinetic)** | **FMOD Studio** | 对本项目的含义 |
|------|--------------------------|-----------------|----------------|
| 确定性适配 | 中立（事件/Switch/RTPC 模型，触发源自定义） | 中立（事件/参数/快照 模型，触发源自定义） | **两者一致**：确定性在集成层，不在引擎 |
| 移动端 runtime 体积 | 较大（~数 MB 原生库 + 较大运行时） | 较小（~1-2 MB 原生库，轻量） | FMOD 更省移动端包体与内存 |
| 移动端性能 | 强，内置精细 voice/CPU/内存预算表 | 强，API 可查 CPU/内存，但预算仪表不如 Wwise 开箱即用 | Wwise 预算执行力更强，FMOD 够用 |
| 自适应音乐 | Music Switch Container + RTPC + Transition 极强 | 参数 + Snapshot + Transition 完整，覆盖本项目需求 | 两者均满足 |
| 空间音频 | 完整 2D/3D、HDR、下游混音 | 完整 2D/3D、快照混音 | 均满足（本项目主要用 2D pan） |
| 授权成本 | Indie 免费（营收 < $20 万）；超阈值 $5000/项目/年 | **完全免费**（无营收阈值，仅付费支持可选） | **FMOD 对早期/独立项目零成本** |
| Unity 集成 | 官方 Wwise Unity 集成（成熟） | 官方 FMOD Unity 集成（成熟、轻量） | 均成熟 |
| 事件命名 | GUID/Event 名（字符串） | Event 名（字符串）+ GUID | 均引擎无关，便于未来迁移 |

### 3.2 确定性适配说明（两者等价）

- 二者都允许"用 Switch 传确定性类别（如法术种类）、用 RTPC 传确定性数值（如战斗强度）"，也允许"在事件内部做 cosmetic 微随机（pitch±5%）"——后者永不回馈模拟，因此**对 desync 零影响**。
- 二者都提供"Event Post"接口（`AkSoundEngine.PostEvent` / `FMODUnity.RuntimeManager.PlayOneShot`），我们统一从**模拟事件队列**调用，而非从渲染/输入层。
- 因此选型**不取决于**确定性，而取决于：① 移动端体积/成本；② 自适应音乐需求；③ 预算执行力。

### 3.3 MVP 推荐：**FMOD Studio**

> **MVP 采用 FMOD Studio**，Wwise 作为规模扩大后的升级路径。

**理由**：
1. **零授权成本**：早期独立项目无营收阈值压力，预算全投入内容。
2. **移动端体积/内存更优**：更小的原生库与运行时，直接利好 §7 严格内存预算。
3. **自适应音乐够用**：FMOD 参数（Parameter）+ 快照（Snapshot）+ 过渡（Transition）完全覆盖 `CombatIntensity` RTPC 动态混音需求（§6）。
4. **轻量 Unity 集成**：与本项目 Unity 6000.3.19f1 兼容，集成成本低。
5. **迁移成本可控**：事件以字符串/逻辑名定义（见 §3.4），若后续需 Wwise 的更强预算仪表/HDR，重制主要在中间件工程侧，游戏侧调用契约不变。

**升级到 Wwise 的触发条件**（供主理人评估）：项目进入 P2+ 需要 AAA 级移动端实时预算仪表、HDR 混音、或复杂 Music Switch 编排时。

### 3.4 事件结构（引擎无关，FMOD MVP 草案）

```
Events/
  Gameplay/
    Cast/                 # Switch: SpellCategory
      Evt_Cast            (Switch=SpellCategory, 由模拟传法术种类)
    Hit/
      Evt_Hit_Player      (Switch=SpellCategory)
      Evt_Knockback
    Death/
      Evt_OOB_Death
      Evt_Boundary_Warn   (RTPC=EdgeProximity 0..1)
    Item/
      Evt_Item_Pickup
      Evt_Item_Drop
      Evt_Item_Telegraph
    Move/
      Evt_Dash
      Evt_Jump
      Evt_Spell_Switch
    Env/
      Evt_Material_Reaction (Switch=ReactionType)
  UI/
    Evt_UI_Click
    Evt_UI_Hover
    Evt_UI_Panel_Open
    Evt_UI_Result_Win
    Evt_UI_Result_Lose
Music/
    Mus_Battle            (RTPC=CombatIntensity 0..1; 多 stem 交叉淡入)

```

> **Switch / RTPC 全部由确定性模拟数据填充**（法术种类来自 `PlayerData.SpellIds[SelectedSpellSlot]`，强度来自存活人数/交火计数），禁止客户端猜测。

## 4. 确定性音频触发模型（Sim → Audio 事件队列）

### 4.1 `SimAudioEvent` 结构（草案，AOT 兼容 blittable 建议）

```csharp
// NoitaCA.Core/Audio/SimAudioEvent.cs  （模拟层产出，纯表现，不进快照/哈希）
public enum SimAudioEventType : byte
{
    SpellCast = 0,        // 施法（含法术类别 Switch）
    PlayerHit = 1,        // 法术命中玩家
    Knockback = 2,        // 玩家被击退
    OOBDeath = 3,         // 出界死亡
    BoundaryWarn = 4,     // 边界警示（进入 15–55px 边缘带）
    ItemPickup = 5,
    ItemDrop = 6,
    ItemTelegraph = 7,    // 道具刷新预告光柱（GDD §7，提前 90 帧）
    Dash = 8,
    Jump = 9,
    SpellSwitch = 10,     // SelectNext/Prev
    MaterialReaction = 11,// 材料反应（水+火→蒸汽 等，CA 模拟确定性产出）
}

public struct SimAudioEvent
{
    public SimAudioEventType Type;
    public int   Frame;            // 确定性逻辑帧号（提交/去重依据）
    public byte  SourcePlayerId;   // 发声主体（用于 2D pan / 归属）
    public int   PosX, PosY;       // 确定性定点位置（Fix64 取整，用于空间化）
    public byte  SwitchParam;      // 如 SpellCategory / ReactionType
    public float RtpcValue;        // 如 EdgeProximity / CombatIntensity
}
```

### 4.2 提交时机（回滚安全）

| 阶段 | 行为 |
|------|------|
| 模拟步进 | `ApplyInputsAndStep(frame)` 内同步 `EnqueueAudio(event)` → 写入 `audioBuffer[frame]`（独立缓冲，不入 `PlayerLayerSnapshot`） |
| 预测期（未确认） | 本地玩家**自身**动作事件可立即提交（自身输入即权威，几乎不回滚）；远端玩家事件**暂不提交** |
| 帧确认 | 收到 `G2C_FrameBatch` 且帧过 INPUT_DELAY(=2) 后，`CommitAudio(frame)` 把 `audioBuffer[frame]` 投递音频引擎 |
| 回滚 | `RollbackTo(mismatch)` 时 `audioBuffer` 中 mismatch..current 帧**整体丢弃**；`ReplayForward` 重跑模拟会**重新产生**这些事件 → 音频与权威模拟逐帧一致 |
| 远端事件 | 远端玩家事件在其输入被服务端确认（写入 `remoteInputs`）后才提交，避免预测错误导致的"幽灵音效" |

> **要点**：音频事件缓冲与快照同生命周期管理，但**不计入状态哈希**——回滚丢弃它不影响确定性校验，只影响听感，而重模拟会重建听感。这是帧同步音频的标准做法。

### 4.3 事件 → 音效映射约束（禁客户端随机）

| 约束 | 允许 | 禁止 |
|------|------|------|
| 触发源 | `SimAudioEvent`（模拟同步产出） | `Update()` 渲染时序、`OnCollisionEnter`（非确定性渲染碰撞）、输入 `GetKeyDown` 直接播声 |
| 类别选择 | 用 `SwitchParam` = `PlayerData.SpellIds[SelectedSpellSlot]` | 客户端猜测"刚按了火"用 `Random.Range` 选音色 |
| 是否发声 | 由事件存在性决定（模拟产生了就播） | `if (Random.value < 0.5f) Play()` |
| 强度/距离 | `RtpcValue` 来自模拟量（边缘距离、存活数） | `Time.deltaTime` 累积触发、浮点抖动触发 |
| 微变体 | 允许引擎内部 pitch±5% 随机化（cosmetic） | 用随机决定"播 A 还是播 B 两种不同语义音" |

## 5. SFX 清单（P0 核心 / P1 扩展）

> **优先级**：`CRIT`(0) > `HIGH`(1) > `MED`(2) > `LOW`(3)。用于 voice stealing 与 music ducking。
> **绑定枚举**：均为 `SimAudioEventType`（见 §4.1）。`ActionFlags` 引用协议规范 §5。

### 5.1 施法（按法术类别区分音色）

施法音靠 **`SpellCategory` Switch** 区分音色，由 `PlayerData.SpellIds[SelectedSpellSlot]` 确定性提供（循环切换模型，ADR §5 / 协议 §4.3）。

| 法术（ADR §7） | 类别 Switch | 音色设计 | 触发事件 | 优先级 | 阶段 |
|----------------|------------|---------|----------|--------|------|
| Bolt 轻弹 (P0) | ProjectileLight | 高频"咻"短促、轻快 | `SpellCast`(Switch=ProjectileLight) | HIGH | P0 |
| Heavy 重弹 (P0) | ProjectileHeavy | 低频"砰"+ 破风，分量感 | `SpellCast`(Switch=ProjectileHeavy) | HIGH | P0 |
| Bomb 炸弹 (P0) | Explosive | 发射"噗" + 另设爆炸命中音（见 5.2） | `SpellCast`(Switch=Explosive) | HIGH | P0 |
| Fire 火球 (P0) | Fire | 呼啸 + 火焰噼啪质感 | `SpellCast`(Switch=Fire) | HIGH | P0 |
| Water 水流术 (P0) | Water | 液体涌动 / 咕噜 | `SpellCast`(Switch=Water) | HIGH | P0 |
| StoneWall 石墙术 (P0) | Earth_Stone | 厚重隆隆 + 砌筑感 | `SpellCast`(Switch=Earth_Stone) | HIGH | P0 |
| Ice 冰锥 (P1) | Ice | 清脆冰晶 shard | `SpellCast`(Switch=Ice) | HIGH | P1 |
| Poison 毒弹 (P1) | Poison | 冒泡、黏腻毒感 | `SpellCast`(Switch=Poison) | HIGH | P1 |
| Acid 酸弹 (P1) | Acid | 腐蚀嘶嘶声 | `SpellCast`(Switch=Acid) | HIGH | P1 |
| Lava 熔岩喷发 (P1) | Lava | 熔岩翻滚轰鸣 | `SpellCast`(Switch=Lava) | HIGH | P1 |
| Shield 护盾泡 (P1) | Shield | 气泡形成 + 微光 | `SpellCast`(Switch=Shield) | HIGH | P1 |
| Blink 闪现 (P1) | Blink | 空间扭曲 whoosh | `SpellCast`(Switch=Blink) | HIGH | P1 |

> 触发条件：`ActionFlags.Attack`(=CastSpell, bit1) 在 `PlayerMovementJob.CastSpell` 中处理，且冷却 + 共享 Mana（100/回复15s，ADR §5）校验通过后，enqueue `SpellCast`。**被 `ValidateInput` 拒绝的无效施法不发声。**

### 5.2 命中 / 击退

| 音效 | 音色设计 | 触发事件 | 优先级 | 阶段 |
|------|---------|----------|--------|------|
| 法术命中玩家 | 用 `SpellCategory` Switch 复用施法材质质感 + 命中冲击层（爆炸类叠 boom） | `PlayerHit`(Switch=SpellCategory) | HIGH | P0 |
| 爆炸命中（Bomb/Lava/Fire+毒） | 分层爆炸：低频 body + 中频 crack + 高频 spark | `PlayerHit`(Switch=Explosive/Lava…) | HIGH | P0 |
| 击退 | 被推时的"嗖+闷响"，强弱随击退力度（来自确定性击退模型） | `Knockback`(RTPC=KnockbackForce) | MED | P0 |

> 命中判定：像素采样确定性命中（GDD §6.2）。击退触发：被命中者处于边缘 15–55px 或受爆炸/重弹推动（ADR §1 击退推人模型，数值见舒衡 v0.2 K 表）。边界警示见 5.3。

### 5.3 出界死亡 / 边界警示

| 音效 | 音色设计 | 触发事件 | 优先级 | 阶段 |
|------|---------|----------|--------|------|
| 出界死亡 | 招牌"坠落/湮灭"强音 + 短促下行动机，标志一名玩家出局 | `OOBDeath` | **CRIT** | P0 |
| 边界警示 | 进入边缘带（15–55px）的上升脉冲音，`RTPC=EdgeProximity` 越近越急 | `BoundaryWarn`(RTPC=EdgeProximity) | HIGH | P0 |

> 触发：玩家像素中心越过边界（GDD §4.2：`y > Height+16` 或 `x ∉ [-16, Width+16]`）→ `OOBDeath`；位置落入边缘带（中心距边界 15–55px）→ `BoundaryWarn`（持续更新 RTPC，离开带则停）。边界红光（UI）与警示音同源触发，强化"出界即死"张力。

### 5.4 道具拾取 / 丢弃 / 预告

| 音效 | 音色设计 | 触发事件 | 优先级 | 阶段 |
|------|---------|----------|--------|------|
| 道具拾取 | 清亮"叮"+ 类别微差（武器/卷轴/药水/护盾/加速/炸弹） | `ItemPickup`(Switch=ItemType) | MED | P0 |
| 道具丢弃 | 闷顿落地声 | `ItemDrop` | MED | P0 |
| 道具预告 | 刷新前 90 帧（GDD §7）的蓄能光柱音，3s 渐强 | `ItemTelegraph`(RTPC=Charge) | LOW | P1 |

> 触发：`ActionFlags.Pickup`(bit4) / `Drop`(bit5) 处理通过 → enqueue；`ItemTelegraph` 由确定性 `SceneScheduler` 在 `TargetFrame-90` 下发时同步产生（所有人同帧，公平）。

### 5.5 冲刺 Dash

| 音效 | 音色设计 | 触发事件 | 优先级 | 阶段 |
|------|---------|----------|--------|------|
| 冲刺 | 短促 directional swoosh（约 0.16s，对齐 ADR §3 爆发耗时），可随 `AimAngle` 做 2D pan | `Dash` | MED | P0 |

> 触发：`ActionFlags.Dash`(bit9) 且冷却（1.8s/54 帧）内未被 `ValidateInput` 拒绝 → enqueue。MVP 无 i-frames（ADR §3），故冲刺音**不含"无敌"提示音**。

### 5.6 UI（菜单 / 结算）——本地触发，非模拟

| 音效 | 触发（本地 UI 层） | 优先级 | 阶段 |
|------|-------------------|--------|------|
| 按钮点击 | `UI_Click`（菜单/暂停/设置本地事件） | LOW | P0 |
| 按钮悬停 | `UI_Hover` | LOW | P1 |
| 面板开启 | `UI_Panel_Open` | LOW | P0 |
| 结算胜 | `UI_Result_Win`（据 `G2C_MatchSettle.Rank`，服务端权威） | MED | P0 |
| 结算负 | `UI_Result_Lose` | MED | P0 |

> **确定性说明**：UI 音属元操作（对齐 UI_UX §2.1：`PauseMenu`/`QuickChat` 不进 `InputPayload`），本地触发**合法且不破坏同步**——它不影响世界状态、不参与哈希。仅"结算胜负"依据服务端权威 `Rank`，避免客户端自行判定名次导致听感分歧。

| 11 | 跳跃 | `Jump` ← `ActionFlags.Jump`(bit0) | 模拟 | MED | P1 |
| 12 | 法术切换 | `SpellSwitch` ← `SelectNext/Prev` | 模拟 | LOW | P1 |
| 13 | 材料反应 | `MaterialReaction` ← CA 反应表 | 模拟 | LOW | P1 |
| 14 | UI 点击/悬停/面板 | 本地 UI 事件 | 本地 | LOW | P0/P1 |
| 15 | 结算胜/负 | `UI_Result_*` ← `G2C_MatchSettle.Rank` | 本地(权威) | MED | P0 |

## 6. 音乐（对战 / 菜单 / 结算）与自适应混音

### 6.1 主题与动机

| 曲目 | 定位 | 设计 |
|------|------|------|
| **Mus_Battle** | 对战 BGM | 驱动型节拍（120–140 BPM），紧张感源自**分层叠加**而非提速；核心动机短小、循环可拼接 |
| **Mus_Menu** | 主菜单/暂停 | 低强度氛围主题，与对战同源动机的"慢速安全版"，建立 IP 辨识 |
| **Mus_Settle** | 结算 | 由胜负 stinger（短促高潮/收敛）接入结算主题 |

### 6.2 自适应混音规则（CombatIntensity RTPC）

| 强度档 | CombatIntensity | 混音状态 | 触发条件（确定性） |
|--------|----------------|----------|-------------------|
| 平静 | 0.0–0.3 | 仅 base 节拍层 | 存活≥3 且近期无交火（近 3s 命中/击退计数=0） |
| 交火 | 0.3–0.7 | base + tension 层（上升琶音淡入） | 近期有命中/击退，或存活≤3 |
| 高潮 | 0.7–1.0 | base + tension + full combat 层（铜管/完整织体） | 存活=2（残局 1v1，GDD §3）或连续击退 |

> `CombatIntensity` 由模拟确定性量推导：**存活玩家数**（递减推高强度）+ **近 N 帧交火事件计数**（命中/击退/死亡）。全部来自 `SimAudioEvent` 统计，非客户端计时器。

### 6.3 触发与切换约束

- **不硬切**：各 stem 用 2–4s 交叉淡入淡出（FMOD Transition / Wwise Music Switch），`CombatIntensity` 阈值跨越时渐变，避免突兀。
- **对战开始/结束**：进入 Battle 状态机 → 播 `Mus_Battle`；进入 `MatchEnded` → 播出战层、延迟接 `Mus_Settle`。状态来自确定性对局状态机。
- **菜单**：主菜单/暂停本地切换（非模拟），合法。
- **结算胜负**：据 `G2C_MatchSettle.Rank`（服务端权威）选胜/负 stinger，避免客户端名次误判。

## 7. 性能预算（移动端，MVP 硬上限）

> 目标机型：中端 Android（骁龙 6xx 级）/ iPhone 11 级。PC 档位为放宽值。

### 7.1 资源上限

| 项 | 移动端 | PC |
|----|--------|-----|
| SFX 数据总包（压缩后） | **≤ 10 MB** | ≤ 40 MB |
| 单条 SFX 时长 | ≤ 1.5 s（短促，降低解码成本） | ≤ 3 s |
| 音乐（流式，不全驻留） | battle 3 + menu 1.5 + settle 1.5 ≈ **≤ 6 MB** | ≤ 20 MB |
| 压缩格式 | SFX: Vorbis 64–96 kbps / ADPCM（极短音）；Music: Vorbis 96–128 kbps 流式 | 同，可更高 |

### 7.2 同时发声数（Voice）

| 项 | 移动端 | PC |
|----|--------|-----|
| 最大真实 voice | **24**（SFX 20 + 音乐 4 stem） | 48 |
| 虚拟 voice / 实例上限 | 64（超则按优先级 steal） | 128 |
| 单事件并发实例上限 | 同种类 ≤ 4（防叠爆） | ≤ 8 |

### 7.3 内存

| 项 | 移动端 | PC |
|----|--------|-----|
| 音频运行时 RAM（含 bank 常驻 + 流式 ring buffer 1–2 MB） | **≤ 16 MB** | ≤ 64 MB |
| 流式缓冲 | 1–2 MB ring | 2–4 MB |

### 7.4 CPU / 采样率

| 项 | 移动端 | PC |
|----|--------|-----|
| 输出采样率 | **32 kHz** | 48 kHz |
| 音频 DSP CPU 占用 | **≤ 6%** 单中端核 | ≤ 10% |
| 输出延迟 | 4–6 帧可接受（不影响确定性，仅听感） | 低延迟优先 |

### 7.5 优先级 Steal / Ducking

- **Voice Stealing**：超出 §7.2 上限时，按优先级 `CRIT > HIGH > MED > LOW` 抢占最低优先级 voice；同优先级抢占最旧实例。
- **Music Ducking**：出现 `HIGH`/`CRIT` 玩法音（命中/死亡/施法密集）时，音乐瞬时压低 **−6 dB**，事件簇结束后 0.5–1s 恢复，保证玩法反馈清晰。
- **爆音防护**：移动端单帧同时提交事件 > 8 条时，合并同类、`CRIT` 必保，其余按优先级丢弃（绝不超出 voice 上限导致卡顿/爆音）。

## 8. 集成接口契约（给端木林 / 客户端）

> 以下为音频集成侧需客户端确认的接口点，建议作为 `NoitaCA.Lockstep` 与音频系统的边界：

1. **事件队列接入点**：在 `ClientLockstepScheduler.ApplyInputsAndStep(frame)` 与 `ReplayForward` 中挂接 `EnqueueAudio(SimAudioEvent)`；在帧确认（过 INPUT_DELAY）后 `CommitAudio(frame)`。回滚时清理 `audioBuffer[mismatch..current]`。
2. **音频缓冲归属**：`audioBuffer` 独立于 `PlayerLayerSnapshot`（不入状态哈希、不入回滚快照状态量）。
3. **内存上限**：移动端音频运行时 RAM ≤ 16 MB、最大 voice 24 —— 需端木林在客户端资源预算表（YooAsset 加载档）中确认并预留。
4. **位置来源**：空间化用的 `PosX/PosY` 取自确定性模拟定点位置（非渲染插值坐标），保证跨端 pan 一致。
5. **禁止调用点**：渲染层 `Update`、非确定性物理回调、输入 `GetKeyDown` 直接播声——统一经 `SimAudioEvent` 或本地 UI 事件通道。
