## 1. 概述与目标

### 1.1 文档目的

补齐 `设计审查_2026-07-07.md` 列出的 P1 缺失文档「结算/结果界面规格」。定义对局结束（最后存活者/队伍胜、出界即死判定后）到返回大厅之间，玩家所见的**结算界面**：触发时机、信息层级、PC/移动双端布局、交互动线、视觉规范、无障碍与既有 HUD/菜单规范的衔接。

设计侧核心问题（来自设计审查）：*「死亡/胜负判定后的 UI 流程、MVP/排名、重开、观战未定义；与边界死亡规则强绑定」*——本文件给出可落地的答案。

### 1.2 设计原则（对齐 `UIUX交互设计.md §1.2` + `美术风格指南.md`）

| 原则 | 在结算界面中的落地 |
|------|------------------|
| **确定性优先** | 展示数据来自帧同步最终 sim 状态（客户端本地确定性派生）或权威 `ResultPayload`（仅 `MmrDelta`）；UI 不引入随机，动画整数帧驱动。 |
| **可读性** | 胜负一眼可读（大横幅 + 状态色 + 符号）；排名按出局顺序倒推，逻辑自解释。 |
| **跨平台一致性** | PC 键鼠与移动触屏「看到的指标、能点的动作」完全一致；仅采集方式不同（鼠标点击 vs 触摸热区）。 |
| **低门槛高上限** | P0 指标（胜负/排名/击杀/存活）零学习成本；P1 深度指标（施法统计/材料改造）满足硬核玩家复盘。 |

### 1.3 文档范围

- ✅ 覆盖：触发时机、排名算法、信息层级（P0/P1 标注）、PC/移动布局、交互（再来一局/返回大厅/查看回放/分享）、视觉规范、无障碍、与 HUD/菜单衔接、走查清单。
- ❌ 不覆盖：网络协议字段编码（以 `协议与序列化规范.md` 为准）、MMR 算法细节（见 `多人联机帧同步对战设计.md §MMR`）、像素模拟。

## 2. 触发时机与结算流程

### 2.1 触发条件（对齐 `游戏设计文档.md §4.2–§4.3`）

结算界面在**对局状态机进入 `MatchEnded`** 时弹出，满足以下任一即判定对局结束：

| 结束条件 | 说明 | 来源 |
|----------|------|------|
| 最后存活者诞生 | `Kill` 模式下仅 1 名玩家未出界 → 该玩家胜，其余按出局顺序定名次 | GDD §4.3 |
| 全员出界 | 所有玩家均已 `PlayerDeath` → 按出局顺序定名次（最后出界者名次最高） | GDD §4.3 |
| 超时/特殊模式收束 | `Bounce`/`Wrap` 模式到达时间上限且 ≥2 人存活 → 按存活者定名次，可能**平局** | GDD §4.1 模式表 |
| 房主/系统终止 | 自建房房主解散、或断线判负兜底（见 §2.4） | `帧同步Netcode设计.md §5.6` |

> **死亡 → 结算的衔接**：单个玩家出界触发 `PlayerDeath` 后，该玩家立即进入**观战**（延迟 1 秒，可发快捷表情，见 GDD §4.2）。结算界面**仅在 `MatchEnded` 时**统一弹出，覆盖所有客户端（含已死亡的观战者）。即：早期出局者观战等待 → 对局收束 → 结算界面替换观战视图。

### 2.2 结算时序（状态机）

```
[对局 Battle]
   │  MatchEnded 触发（最后存活 / 全员出界 / 超时）
   ▼
[① 数据汇聚]  服务端下发 G2C_MatchSettle（精简 SettlePlayer[] + MmrDelta）；
   │          客户端已持有最终 sim 状态，富字段本地派生（见 §3.3）
   ▼
[② 揭晓动画]  整数帧驱动（见 §6.4）：横幅滑入 → 排名逐行点亮（约 5s 展示节拍，GDD §3「结算 Result 5s」）
   │          此阶段按钮可点击，但视觉重心在「揭晓」
   ▼
[③ 可交互态]  按钮全亮；开始可选「挂机自动返回」倒计时（默认 30s，设置可调；到时自动返回大厅）
   │
   ├─[再来一局]──▶ 按模式重开（§5.2）
   ├─[返回大厅]──▶ 主菜单/大厅（UX §6.1 流：Result → A）
   ├─[查看回放]──▶ 条件显示（§5.4，依赖 netcode 回放能力）
   └─[分享战绩]──▶ 生成战绩卡（§5.5）
```

> **「5s」的解读**：GDD §3 核心循环的「结算 Result 5s」指**揭晓动画节拍**（约 5 秒把胜负与排名逐步呈现），**不是强制自动返回**。强制自动返回默认关闭，仅提供「挂机 N 秒后自动返回大厅」（默认 30s，设置项），用于清理掉线/挂机玩家。按钮在揭晓动画期间即可点击。

### 2.3 排名算法（按出局顺序倒推 · 确定性）

> **计算归属（已与客户端确认 · v1.2 修正）**：帧同步下客户端在 `MatchEnded` 时已持有最终 sim 状态，故 `Rank / Kills / Deaths / IsWinner / IsDraw / SurvivalMs / MatchDurationMs / MaterialModifiedPixels` 等**确定性量由客户端从本地 sim + 房间名册本地派生**（各客户端结果一致，符合「数据来自权威」）；仅 `MmrDelta`（非确定性）由**服务端权威下发**。wire 形态为精简 `SettlePlayer`（见 §3.3 与 `协议与序列化规范.md §6.1`），非全量推送——避免实现时误以为要等服务端推送富字段。

名次由**出局顺序**决定（GDD §4.3「多人同时出界按出局顺序定名次」）。算法须为**确定性全序**（客户端从本地 sim 派生，结果各端一致，是帧同步公平性的延伸）：

```
输入：本地 sim 最终状态（每位玩家含 IsAliveAtEnd、EliminationFrame、Kills、SlotIndex）
输出：Rank[1..N]，以及 IsWinner / IsDraw 标记

1) 分组：Survivors = {IsAliveAtEnd == true}；Eliminated = 其余
2) 名次分配：
   a) Survivors 名次最前。若 Survivors 仅 1 人 → Rank=1，IsWinner=true。
      若 Survivors ≥2 人 → 共享 Rank=1，IsDraw=true（平局，见 §2.4）。
   b) Eliminated 按 EliminationFrame 降序排列（越晚出局名次越高）。
   c) 并列裁决（确定性 tie-break，逐级生效）：
      - 同 EliminationFrame → 按模拟内 PlayerDeath 事件处理的确定性顺序（锁步内顺序全客户端一致）
      - 仍并列 → SurvivalMs 较长者优先
      - 仍并列 → SlotIndex 较小者优先（最终确定性裁决，消除一切歧义）
3) 连续名次：Eliminated 区紧接 Survivors 之后按 (b)(c) 顺序连续编号。
```

- `EliminationFrame`：该玩家 `PlayerDeath` 触发的逻辑 tick（来自本地 sim，确定性）。
- `SurvivalMs`：对局开始（或出生）到 `PlayerDeath` 帧的时长（由 `TotalFrames` 推导）；胜者为整局时长。
- 击杀归属遵循 GDD §11 Q3 待拍板方向：「环境/生物致死按最后接触玩家归属」——**Kills 统计依赖该归属规则落地**（见 §9.1）。

### 2.4 平局（Draw）判定

| 场景 | 处理 |
|------|------|
| `Bounce`/`Wrap` 超时且 ≥2 存活 | 存活者共享 Rank=1，`IsDraw=true`，结算界面本地玩家显示**平局色（中性灰）** |
| 终局多人同帧出界且 tie-break 后仍无法区分（极端） | 共享最高名次，`IsDraw=true` |
| `Kill` 模式正常收束 | 必有唯一最后存活者 → 无平局（除非断线判负兜底造成并列，按 §2.1 末行处理） |

> 平局时**不展示「胜利/失败」横幅**，改为中性「平局」横幅 + ＝/≈ 符号（视觉见 §6.2）。

## 3. 信息架构与指标层级

### 3.1 信息优先级（核心行动优先）

```
P0 必读（一眼）：[胜负结果横幅] > [我的名次] > [击杀 K] > [存活时长]
P0 必做（动作）：[再来一局] / [返回大厅]
P1 复盘（可展开）：[施法统计] / [材料改造量] / [助攻 A] / [查看回放] / [分享战绩]
次要：MMR 变化（排位专用，P0 for 排位）、房间/模式标识
```

### 3.2 指标清单（标注 P0 / P1）

| 指标 | 优先级 | 说明 | 取值来源 |
|------|--------|------|----------|
| **胜负结果**（胜/负/平） | **P0** | 本地玩家横幅 + 状态色 + 符号 | 客户端派生（Rank + Players[]）；符号见 §6.2 |
| **排名**（按出局顺序倒推） | **P0** | 全玩家名次榜，本地高亮 | 客户端从 sim 派生（§2.3） |
| **击杀数 K** | **P0** | 直接/归属击杀 | 客户端从 sim 派生 |
| **存活时长** | **P0** | 出生到出局/终局 | 客户端从 `TotalFrames` 派生 |
| **死亡数 D** | **P0** | 胜者=0，其余=1 | 客户端从 sim 派生 |
| **MMR 变化**（排位） | **P0（仅排位）** | 名次→ELO 变体（K=32, Base=1000） | **服务端下发**（非确定性） |
| **再来一局 / 返回大厅** | **P0** | 核心动作 | 交互 §5 |
| **施法统计**（总次数 + 分元素） | **P1** | 火/冰/毒/酸/投掷等施法次数 | 客户端从 sim 派生 |
| **材料改造量** | **P1** | 玩家引动/转化的像素数（GDD §9「可选」） | 客户端从 sim 派生 |
| **助攻 A** | **P1** | 参与击杀但非终结者（需归属规则） | 客户端从 sim 派生（§9.1） |
| **查看回放** | **P1（条件）** | 依赖 netcode 回放能力（§9.2） | `ReplayAvailable`（服务端） |
| **分享战绩** | **P1** | 生成战绩卡分享 | 交互 §5.5 |

> **MVP 定义**：P0 = 首版 MVP 必做；P1 = 首版可后置或条件开放。材料改造量在 GDD §9 明确为「可选」，列为 P1。
> **来源总则（v1.2）**：除 `MmrDelta` 由服务端权威下发外，其余确定性指标均由**客户端从本地 sim 状态 + 房间名册本地派生**（各客户端一致），不依赖服务端全量推送。

### 3.3 字段定义与取值来源（数据契约草案）

> 供客户端（端木林）与协议文档对齐。最终字段以 `协议与序列化规范.md` 为准；此处定义 **UI 侧的「逻辑/展示模型」**与字段来源。承载消息 `G2C_MatchSettle`（S→C，protobuf 低频，不占 30Hz 热路径，不触发 ProtocolVersion bump）已落 `协议与序列化规范.md §6.1`（ADR §8）；精简 wire 形态 `SettlePlayer` 见该文档，本结构为 UI 消费形态。

```csharp
// UI 展示模型（客户端本地组装，非 wire 全量）
struct ResultPayload        // 由 G2C_MatchSettle + 本地 sim/名册派生
{
    uint32          MatchId;
    ArenaConfig     Arena;          // 复用主架构 §4.4（Width/Height/Seed/BoundaryMode/MaxSpeed），「再来一局」直接复用
    uint32          TotalFrames;    // 整局逻辑帧数（→ MatchDurationMs 派生）
    bool            IsRanked;
    PlayerResult[]  Players;        // 长度 = 房间人数；富字段客户端派生
    bool            ReplayAvailable;// 依赖 netcode 是否实现输入流录制（§9.2，当前=false）
}

struct PlayerResult        // UI 消费形态
{
    uint32  PlayerId;
    string  DisplayName;            // 房间名册按 PlayerId 查得（不在 wire）
    uint8   Rank;                   // 客户端从 sim 派生（服务端亦给 YourselfRank 便捷项）
    bool    IsWinner;               // 由 Rank + Players[] 推导（不在 wire）
    bool    IsDraw;                 // 由 Rank + Players[] 推导（不在 wire）
    uint16  Kills;                  // K（客户端从 sim 派生）
    uint16  Deaths;                 // D（0 或 1，派生）
    uint16  Assists;                // A（P1，待归属规则，派生）
    uint32  SurvivalMs;             // 由 TotalFrames 推导（派生）
    uint32  SpellsCast;             // P1 总施法（派生）
    // 分元素施法 map（P1，可选）：Element -> count（派生）
    uint32  MaterialModifiedPixels; // P1 材料改造量（派生）
    uint32  EliminationFrame;       // 0 = 存活到终局（派生）
    int32   MmrDelta;               // 仅此非确定性量由服务端权威下发（在 wire）
}
```

**wire vs 派生说明（v1.2 同步 · 与 `协议与序列化规范.md §6.1` 对齐）**：`PlayerResult` 是**展示模型**（UI 消费形态，字段丰富）；实际 wire（`SettlePlayer`）为**精简形态**——帧同步下客户端在 `MatchEnded` 时已持有最终 sim 状态，富字段可本地确定性派生：

- `Rank / Kills / Deaths / Assists / SurvivalMs / SpellsCast / MaterialModifiedPixels / EliminationFrame`：客户端从**本地 sim 状态**派生；
- `DisplayName`：房间名册按 `PlayerId` 查得（**不在 wire**）；
- `IsWinner / IsDraw`：由 `Rank` 与 `Players[]` 推导（**不在 wire**）；
- `MatchDurationMs`：由 `TotalFrames` 推导（**不在 wire**）；
- `MmrDelta`：**仅此非确定性量由服务端权威下发**（在 wire）；
- `YourselfRank`：服务端在 `G2C_MatchSettle` 内另给（便捷项），客户端亦可按本地 `PlayerId` 在 `Players[]` 定位，二选一。

> 实现要点：**勿等待服务端全量推送富字段**；只需拿到精简 `SettlePlayer`（含 `PlayerId` + `MmrDelta` 等必要项）+ 本地 sim/名册即可组装出本 `PlayerResult`。`IsLocalPlayer` 由客户端比对自身 `PlayerId` 决定，不进 wire。

### 3.4 模式差异

| 模式 | 排名语义 | 额外展示 |
|------|----------|----------|
| 自建房 Custom | 按出局顺序；可同房间重开 | 房间号/房主标识 |
| 快速匹配 Quick | 同上 | 无 MMR |
| 排位赛 Ranked | 同上 + ELO | **MMR 变化（P0，服务端下发）**、禁对手文本（GDD §4.1） |

## 4. 布局

### 4.1 横竖屏策略（对齐 `UIUX交互设计.md §7.1`）

- **强制横屏（Landscape Locked）**：竞技场为宽幅像素世界，竖屏不可用。启动时/进入对局即 `Screen.orientation = Landscape`，并锁定。
- **竖屏检测 → 旋转门（非重排）**：若设备处于竖屏，结算界面**不单独做竖屏布局**，而是显示全屏「请旋转设备」遮罩（像素旋转箭头图标），待旋转为横屏后正常显示。理由：与 HUD/对局一致，避免为小众姿态额外维护一套布局。

```
竖屏门（portrait 时全屏遮罩，非结算内容）：
┌──────────────┐
│   ⟳ 图标      │
│ 请将设备横置  │
│ （竞技场为宽幅│
│   像素世界）  │
└──────────────┘
```

### 5.1 按钮与动作映射（PC / 移动一致语义）

### 5.2 再来一局（Play Again）

按模式差异化（对齐 GDD §4.1 + UX §6.1 流）：

| 模式 | 行为 |
|------|------|
| 自建房 Custom | 房主重开**同一房间**（复用 `ArenaConfig`：Seed/Width/Height/BoundaryMode/玩家列表），不返回大厅。非房主随房主。 |
| 快速匹配 Quick | 重新进入匹配队列（相近 MMR 桶，见架构文档 §匹配），到齐开局。 |
| 排位赛 Ranked | **不提供即时重开**（防 MMR 滥用/SBMM 抖动），按钮替换为「返回大厅 → 重新匹配」。 |

- 点击后展示 1–2s 确定性加载过渡（对齐 GDD §3「加载竞技场 <3s」），禁用按钮防连点。

### 5.3 返回大厅（Return to Lobby）

- 对齐 UX §6.1 菜单流：`对局 Battle → 结算 Result → 主菜单/大厅`。
- 立即释放对局资源；若有挂机自动返回倒计时进行中，手动点击优先。

### 5.4 查看回放（View Replay · 条件）

- **依赖声明（重要）**：当前 `帧同步Netcode设计.md` 仅定义了**断线重连**（§5，全量快照 + 增量重放），**未定义「输入流录制 + 回放播放」能力**。帧同步确定性（30Hz `InputPayload` + 初始 Seed）在原理上**支持**回放（录输入→重模拟），但该特性**尚未在 netcode 中落地**（详见 §9.2，已与客户端确认 `InputRingBuffer` 非整局录制器）。
- **UI 处理**：按钮**条件显示**——仅当 `ReplayAvailable == true` 时渲染；否则隐藏或置灰（禁用态用暗灰 `0x514A`，见 §6.2）。置灰时 tooltip/文案：「回放功能即将上线」。
- **若实现**，交互流：点击 → 进入独立 `ReplayView` 状态（复用对局渲染 + 整数帧回放控制：播放/暂停/倍速，倍速仍走确定性步进，禁浮点）→ 退出回放回到结算或大厅。
- **标记**：P1，且为**阻塞依赖项**（见 §9.2），不阻塞 MVP 首版（按钮默认隐藏）。

### 5.5 分享战绩（Share）

- **生成战绩卡**：客户端由权威数据本地合成一张图片（所有客户端数据一致 → 卡片内容公平、无信息差）。
- **视觉约束**：战绩卡仍沿用 §3 调色板 + 像素字体（与游戏内 UI 同源，品牌一致；不引入板外色，遵守 ART §8 checklist）。
- **平台分发**：
  - 移动：调用原生分享面板（Android `ACTION_SEND` / iOS `UIActivityViewController`），可分享图片或带图的文本。
  - PC：保存 PNG 到剪贴板 / 本地文件，或调用系统分享。
- **内容**：名次、胜负符号、K/D/A、存活时长、模式标识（不含对手文本于排位，GDD §4.1）。

## 6. 视觉规范（对齐 `美术风格指南.md` + RGB565）

### 6.1 取色源与禁用项（强约束）

- **渲染路径（已与客户端确认）**：结算界面为 **UI 叠加层**（与 HUD 同源，Canvas/UGUI 或 UI Toolkit），**非 in-world 像素通道**。叠加层虽是 32 位画布，但我们只涂「调色板展开色」（hex→Color32，公式见 ART §3 line 79 / 本文件 §6.2），不使用板外色、不挂 bloom、不用渐变。若未来要做 in-world 像素结算牌，需走 `PixelWorldRenderer` 路径，属另一档 scope，单独排期。
- **唯一取色源**：`美术风格指南.md §3` 主调色板（12 材料 RGB565）。UI 颜色**必须**从该板取 16 位值，**禁止引入板外色**（ART §8 checklist）。
- **禁用**（破坏确定性，一律禁止）：
  - 后处理随机抖色 / 噪声扰动；
  - 每帧基于 `Time` 的色相偏移；
  - 浮点后处理调色（bloom 等）；
  - 超出 16 位的 HDR / 32 位色。
- **允许**：固定 16 位调色板、整数帧驱动的 Telegraph 式动画、`100ms` 确定性线性插值（ART §2.3）。

### 6.2 胜利 / 失败 / 平局 状态色（本文件核心新增）

从 §3 板内取色，复用 ART §7 / UX §5.4 既有状态色语义（安全=冰青、危险=火橙、禁用=暗灰），保证与 HUD 一致：

| 状态 | RGB565 hex | R,G,B(0-31/0-63/0-31) | 展开 Color32（R/G/B 各补位到 8 位，见 ART §3 line 79） | 符号 | 语义依据 |
|------|-----------|----------------------|------------------------------------------|------|----------|
| **胜利 Win** | `0x73FF` | 14,31,31 | (115,125,255) 冰青 | ★ / ✓ / 像素皇冠 | 复用「安全」语义：你存活到最后 |
| **失败 Lose** | `0xF900` | 31,8,0 | (255,32,0) 火橙红 | ✗ / 像素骷髅 | 复用「危险」语义：你出界即死 |
| **平局 Draw** | `0xA294` | 20,20,20 | (160,160,160) 中性灰 | ＝ / ≈ | 复用「禁用/中性」语义：无胜负 |

> - 强调色（按钮高亮/描边）亦用 `0xF900` 火橙（ART §7「强调色取火橙」）。
> - 文本亮色用 `0xB3DF` 蒸汽近白（22,30,31）；UI 面板底用 `0xA294` 石 / `0x9264` 烟 中性暗色。
> - **色相优先级**：一律以 **hex 为准**（ART §3 声明 hex 为权威 `DefaultColor`）。若 ART 文档中该色的十进制标值与 hex 存在舍入差，以 hex + 上述展开公式得到的 Color32 为渲染依据。
> - **已知 ART 笔误**：`美术风格指南.md §3 line 77` 将 `0x514A` 标注为 (10,10,10)，实解 R5G6B5 为 (10,5,10)。按本规则以 hex + 展开公式为准（客户端执行无碍）；**建议 ART 维护人修正该十进制标注**以免误导。
> - **色盲友好**：胜(青)/负(橙)/平(灰) 三色在红绿色盲（最普遍）与蓝黄色盲下均**可区分**（青 vs 橙为已知色盲安全对），且叠加**符号**（★/✗/＝）双重编码（见 §7.1）。

### 6.3 字体与图标

- **字体**：中文点阵像素字体，英文像素字体（ART §7）；名次数字用等宽像素数字，保证对齐。
  - **资源依赖（owner 待指派）**：像素字体资产（中文点阵 + 等宽像素数字）须由美术/TA 经 **YooAsset** 提供；若资产未到位，结算 UI 将退化为非像素字体，破坏视觉一致性。已列入 §10 走查清单与跨团队依赖（见 §9.3 Q5）。
- **图标**（ART §6 风格，单色+描边，板内取色）：
  - 击杀 ⚔、存活 ⏱、胜利 ★、失败 ✗、平局 ＝、材料 ▦、法术 ✷。
  - 元素法术图标复用 HUD 几何符号（火=三角、冰=六边形、毒=水滴，ART §6）。
- **玩家标识**：排名/房主区分沿用 ART §6「玩家像素小人 4–8 色」；本地玩家行加冰青描边高亮。

### 6.4 动效（整数帧驱动 · 禁随机抖动）

| 动效 | 实现 | 约束 |
|------|------|------|
| 横幅滑入 | 整数帧线性插值（帧计数 0→N），非 `Time.deltaTime` | 确定性，所有客户端同步 |
| 排名逐行点亮 | 按名次顺序，每行间隔固定整数帧点亮（如每 6 帧一行） | 揭晓节拍 ≈5s（GDD §3） |
| 数值滚动（K/存活） | 整数帧递增到目标值 | 禁用浮点缓动 |
| 按钮 hover/按下 | 描边色变（板内色），无缩放抖动 | 无随机 |
| 战绩卡生成 | 静态合成，无动效 | — |

> **渐变禁令**：禁止在两板内色之间做**渐变过渡**（会产出非板内中间色，违反 ART §2.2 RGB565 约束）。改用**离散状态切换**或**整数帧位置插值**（本表已实现，无问题）。例如胜负横幅颜色切换用状态跳变，不用色相 tween。
> **确定性红线**：任何依赖 `Time.deltaTime` 或随机抖色的过渡 → 破坏帧同步确定性，一律禁止（ART §2.2、UX §9 红线）。

### 6.5 与 HUD 视觉 token 复用

结算界面直接复用 HUD 的设计 token（UX §5），保证「对局内 → 结算」视觉连续：

| Token | 来源 | 结算用法 |
|-------|------|----------|
| 状态色（危险/安全/禁用） | UX §5.4 / ART §7 | 胜负横幅、排名胜负标记 |
| HP/击杀文本样式 | UX §5.1 | 战绩区 K/D/A、击杀数 |
| 边界警示色 | UX §5.1 | 失败横幅呼应「出界即死」语义 |
| 像素字体/图标 | ART §6–§7 | 全局 |

## 8. 与 HUD / 菜单规范的衔接

### 8.1 设计 token 复用（见 §6.5）
