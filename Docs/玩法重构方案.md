- 5 个玩法程序集（Core / Simulation / Renderer / Gameplay / Editor）的详细内容

### 1.5 决策 1 论证（ADR 风格）

> **背景**：现有玩法逻辑基于 MonoBehaviour + 托管数组 + `System.Random` + `float`，无法满足帧同步确定性要求（见 §2.3 五大问题）。需重构以支持：bit 级跨平台确定性、Burst 并行、0 GC 热路径。决策 1 选定"全量 DOTS 重构"，以下为方案比选记录。

**考虑的方案**：

| 方案 | 描述 | 风险 | 收益 |
|------|------|------|------|
| **A. 全量 DOTS 重构（一次性）** | 一次性引入 ISystem/IJobChunk + 定点数，PixelGridAdapter 桥接过渡 | 编译中断风险高；工期长（6 阶段）；中间无完整可验证状态 | 无技术债；确定性从头建立；架构干净 |
| **B. 渐进式 3 阶段** | 先定点化 → 再 Burst 化 → 最后 DOTS 化，每阶段保留旧路径 | 双系统维护成本高；阶段间数据转换开销 | 可回滚；每步可独立验证；风险分散 |
| **C. 双轨并行（新旧共存）** | 新 DOTS 系统与旧 MonoBehaviour 同时运行，AB 验证 | 逻辑重复；两套数据流需同步，易 desync | 可线上 AB 验证；回滚成本最低 |

**决策**：选 **A**（全量 DOTS 重构）。

**理由**：
1. 已有 `PixelGridAdapter` 桥接（§4.1.6）降低编译中断风险——未迁移代码通过 Adapter 继续访问像素数据，每阶段结束保证编译通过。
2. 帧同步要求从头确定性。渐进式（B）保留旧 `float`/`System.Random` 代码会持续引入非确定性，阶段间状态哈希无法对齐，反而增加调试成本。
3. 6 阶段实施路线（§10）每阶段设有验证门（编译通过 + 视觉一致 + 确定性回归），保证可回退；并非"一次性大爆炸"。
4. 双轨并行（C）的两套数据流同步本身是 desync 源头，与帧同步目标矛盾。

**Consequences**：
- *正面*：架构干净，无技术债；确定性从 Phase 1 起建立；Burst 覆盖率可达 100%。
- *负面*：Phase 间编译压力大（需 PixelGridAdapter 桥接未迁移代码）；必须严格按阶段顺序推进，不可跳跃；Phase 1-5 期间存在新旧代码共存窗口。

### 2.2 关键文件问题清单

| 文件 | 行数 | 问题 | 影响 |
|------|------|------|------|
| `PixelGrid.cs` | 807 | `Pixel[,]` 托管数组、`bool[,]` 活跃矩阵、`List<Vector2Int>` 频繁增删 | GC 压力大、无法 Burst、无法并行 |
| `Pixel.cs` | - | struct 含 `bool` 字段（`IsActive` / `IsUpdated`），非 blittable | 无法放入 NativeArray、Burst 失败 |
| `PixelSimulation.cs` | 89 | `System.Random` 非确定性 | 多平台/多客户端结果发散 |
| `MovementSystem.cs` | - | 直接 mutate `PixelGrid.Pixels` | 无并行安全、无快照能力 |
| `InteractionSystem.cs` | - | 同上 + 温度计算用 `Math.Sqrt` 浮点 | 浮点跨平台发散 |
| `PlayerController.cs` | 858 | 输入 + 碰撞 + 动画 + sprite 生成 + 血量 + 施法全混 | 单文件难维护、无法热重载测试 |
| `SpellController.cs` | 522 | `List<PixelProjectile>` 每帧 Clear/Add | GC 压力 + 无法 Burst |
| `PixelCreature.cs` | 529 | `List` 频繁 Clear/Add | 同上 |
| `PixelWorldBootstrap.cs` | 1135 | 世界生成 + 相机 + 后处理 + 玩家 + 生物 + 装备全混 | 单文件 1135 行、职责模糊 |
| `InputController.cs` | 283 | 鼠标绘制 + 材质热键 + 缩放混合 | 无法分离玩法输入 vs 调试输入 |
| `PixelWorldRenderer.cs` | - | 主线程 `Texture2D.SetPixels32` + 从 `PixelGrid` 读 `Color32[]` | 主线程瓶颈 |
| `Demo/StressTest*.cs` | - | `Debug.Log($"...")` 字符串插值 | 每帧 GC |

### 2.3 5 大问题总结

1. **数据布局非 DOTS 友好**：`Pixel[,]` 托管数组 + `bool` 字段 + `List<T>` 热路径分配，无法 Burst、无法并行、无法快照
2. **数学非确定性**：`System.Random` + `float` + `Math.Sqrt`，多平台/多客户端结果发散，无法支撑帧同步
3. **上帝类横行**：`PlayerController`（858 行）/ `PixelWorldBootstrap`（1135 行）/ `SpellController`（522 行）/ `PixelCreature`（529 行）单文件混合多职责
4. **主线程渲染瓶颈**：`PixelWorldRenderer` 在主线程 `SetPixels32` + 读 `Color32[]`，256×256 像素每帧 256KB 拷贝
5. **程序集边界缺失**：全部代码在 Assembly-CSharp，无 asmdef 隔离，无法独立编译/测试/热更

#### 4.1.3 定点数 Fix64

> **本节为权威实现**，FRAME_SYNC §8.4 已添加反向引用指向本节。

```csharp
// 伪代码，仅作设计参考
// AOT/FixedPoint/Fix64.cs
// Q31.32 格式, 覆盖 ±2.1e9, 精度 2.3e-10
// 引用 multiplayer-frame-sync-design.md §4.2.1
//
// 【范围限制声明】为避免 64 位乘法溢出，Fix64 有效值范围限制为 ±2^15（±32768）。
// 像素坐标/温度/速度等玩法量级均远小于此阈值，不构成实际约束。
// 乘法采用 double 中间结果：在 Burst FloatMode.Strict 下 double 运算跨平台一致。
// 注：double 53 位精度无法精确表示 2^94 乘积，会有低位精度损失；
// 但 Burst FloatMode.Strict 下 double 运算跨平台一致，不影响确定性。
// 范围限制 ±2^15 已确保 |raw| ≤ 2^47，乘积 ≤ 2^94 在 double 表示范围内（虽精度损失）。
public readonly struct Fix64 : IEquatable<Fix64>, IComparable<Fix64>
{
    public const long One = 1L << 32;
    public const long Half = One >> 1;
    public const double MaxValue = 32768.0;  // ±2^15 范围限制
    private readonly long _rawValue;

    public Fix64(long rawValue) => _rawValue = rawValue;

    [MethodImpl(MethodImplOptions.AggressiveInlining)]
    public static Fix64 FromRaw(long raw) => new Fix64(raw);

    [MethodImpl(MethodImplOptions.AggressiveInlining)]
    public static Fix64 operator +(Fix64 a, Fix64 b) => new Fix64(a._rawValue + b._rawValue);

    [MethodImpl(MethodImplOptions.AggressiveInlining)]
    public static Fix64 operator -(Fix64 a, Fix64 b) => new Fix64(a._rawValue - b._rawValue);

    [MethodImpl(MethodImplOptions.AggressiveInlining)]
    public static Fix64 operator *(Fix64 a, Fix64 b)
    {
        // 使用 double 中间结果避免 64 位溢出（|a|,|b| ≤ 2^15 时 raw 积 ≤ 2^94，double 53 位精度有损但不影响确定性）
        // Burst FloatMode.Strict 下 double 运算跨平台一致
        long result = (long)((double)a._rawValue * (double)b._rawValue / (double)One);
        return new Fix64(result);
    }

    [MethodImpl(MethodImplOptions.AggressiveInlining)]
    public static Fix64 operator /(Fix64 a, Fix64 b)
    {
        // 先把被除数放大到 64 位范围再除，避免 <<32 丢高位
        double result = (double)a._rawValue / (double)b._rawValue;
        return new Fix64((long)(result * (double)One));
    }

    // Sqrt / Sin / Cos / Atan2 使用查表 + 牛顿迭代(详见 Fix64Math.cs)
}

// AOT/FixedPoint/Fix64Vec2.cs
public struct Fix64Vec2
{

    public Fix64 Y;
}
```

**对 §8.3 的 CombinePixels 修正**（issue 2）：§8.3 原 `CombinePixels` 仅哈希 `PixelData` 的 4 个字段（MaterialType / Temperature / Density / Flags），遗漏 VelocityX / VelocityY / Lifetime / Color / FallingFrames 共 5 个字段，导致状态哈希无法覆盖完整像素状态。本文档修正为遍历 `PixelData` 全部 **9 个字段**：

```csharp
// 伪代码，仅作设计参考 —— 本文档 CombinePixels 已包含 Seed 初始化与最终 mix（c2 乘法 + rotl13 + h*5+0xe6546b64），是完整实现，与 FRAME_SYNC §8.3 局部 hash 函数不同。FRAME_SYNC §8.3 已添加反向引用指向本文档修正版。
private static uint CombinePixels(NativeArray<PixelData> data, int start)
{
    // PixelData 全部 9 字段均纳入哈希(原 §8.3 仅 4 字段, 漏 5 项)
    uint h = Seed;
    for (int i = 0; i < 4 && start + i < data.Length; i++)
    {
        ref var p = ref data[start + i];
        // Round 2 issue 修复: short 字段(Temperature/VelocityX/VelocityY/Lifetime)
        // 直接 cast 成 uint 会符号扩展至 32 位, 当值<0 时高位 0xFF 会污染相邻字段位域。
        // 必须用 & 0xFFFF 掩码截断至 16 位, 确保每字段只占其声明位数。
        h ^= (uint)(p.MaterialType) | (((uint)(p.Temperature) & 0xFFFF) << 8) | ((uint)(p.Density) << 24);
        h ^= (uint)(p.Flags)        | ((uint)(p.FallingFrames) << 8) | ((uint)(p.Color) << 16);
        h ^= ((uint)(p.VelocityX) & 0xFFFF) | (((uint)(p.VelocityY) & 0xFFFF) << 16);
        h ^= (uint)(p.Lifetime) & 0xFFFF;
        // 9 字段: MaterialType, Temperature, Density, Flags, FallingFrames,
        //         Color, VelocityX, VelocityY, Lifetime
        h *= 0xcc9e2d51;                          // c1
        h = (h << 15) | (h >> 17);                // rotl 15
        h *= 0x1b873593;                          // c2 (Round 4 修复: 原 §4.1.5 漏此三步)
        h = (h << 13) | (h >> 19);                // rotl 13 (Round 4 修复)
        h = h * 5 + 0xe6546b64;                   // 最终 mix (Round 4 修复)
    }
    return h;
}
```

## 5. Bootstrap 拆分方案

> **用户决策 4**：拆为 4 个职责单一的 MB。
> 现状 `PixelWorldBootstrap.cs` 1135 行上帝类，混合世界生成 + 相机 + 后处理 + 玩家 + 生物 + 装备。

### 5.1 4 个 MB 职责表

| MonoBehaviour | 职责 | 输入 | 输出 | 引用程序集 |
|--------------|------|------|------|-----------|
| `WorldBuilder` | 纯世界生成，确定性 RNG | `ArenaConfig{Width,Height,Seed,Mode,MaxSpeed}（详见 multiplayer-design §4.4）` | ECS 像素实体 + 玩家/生物出生点 | Simulation + Gameplay |
| `CameraRig` | 相机跟随 + 缩放 | 玩家位置 + 输入缩放 | Camera.transform | Renderer |
| `ArtDirector` | 后处理 + 灯光 + 背景 | 配置 | Volume / Light / Backdrop | Renderer |
| `GameBootstrap` | 组装入口，UniTask 异步编排 | 场景加载触发 | 调用上述 3 个 MB | 全部 5 个 |

> **程序集归属说明**：4 个 Bootstrap MB **不在** 5 个 `NoitaCA.*` asmdef 内（避免 `AOT` 反向依赖 `Simulation`/`Renderer`/`Gameplay`），而是保留在 **Assembly-CSharp**（位于 `AOT/Bootstrap/` 目录，仍属 AOT 范畴）。这是"组合根"（composition root）模式：库代码在 asmdef，组装入口在默认程序集，可自由引用全部 asmdef 而不破坏依赖方向。

### 5.2 拆分前后代码量对比

| 文件 | 拆分前行数 | 拆分后 | 估计行数 |
|------|-----------|--------|---------|
| `PixelWorldBootstrap.cs` | 1135 | `WorldBuilder.cs` | ~400 |
| - | - | `CameraRig.cs` | ~120 |
| - | - | `ArtDirector.cs` | ~200（含 `PixelWorldBackdrop` / `PixelWorldRenderSettings` 合并） |
| - | - | `GameBootstrap.cs` | ~180（UniTask 异步编排） |
| **总计** | **1135** | **4 文件** | **~900** |

**收益**：
- 单文件最大 400 行（原 1135 行）
- 职责单一，可独立测试
- `WorldBuilder` 可独立替换（不同竞技场地图）
- `ArtDirector` 可独立调整（视觉风格迭代）

### 5.4 GameBootstrap 用 UniTask 异步加载流程

> **线程约束说明**（issue 11）：DOTS `EntityManager.CreateEntity` / `SetComponentData` 等 ECS 写入操作**必须主线程执行**（违反则抛 `InvalidOperationException`）。仅 ECS 无关的数据生成（噪声计算 / 地形算法 / 资源解析）可切线程池，且在回到主线程后再调用 EntityManager。

```csharp
// 伪代码，仅作设计参考
// AOT/Bootstrap/GameBootstrap.cs (Assembly-CSharp, 组合根)
using Cysharp.Threading.Tasks;
using UnityEngine;

public sealed class GameBootstrap : MonoBehaviour
{
    [SerializeField] private WorldBuilder _worldBuilder;
    [SerializeField] private CameraRig _cameraRig;
    [SerializeField] private ArtDirector _artDirector;
    [SerializeField] private PixelWorldRenderer _renderer;
    [SerializeField] private ArenaConfig _arenaConfig;  // Round 2 issue 修复: 竞技场配置(Width/Height/Seed/Mode/MaxSpeed, 详见 multiplayer-design §4.4)

    private async UniTaskVoid Start()
    {
        var ct = this.GetCancellationTokenOnDestroy();
        try
        {
            await LoadAssetsAsync(ct);              // YooAsset 资源
            await _worldBuilder.BuildWorldAsync(_arenaConfig, ct); // ECS 世界生成(见下方 WorldBuilder)
            await _artDirector.ApplyAsync(ct);       // 后处理
            _cameraRig.Initialize(/* playerEntity */);
            _renderer.Initialize();
        }
        catch (OperationCanceledException) { /* 取消, 不报错 */ }
    }

    private async UniTask LoadAssetsAsync(CancellationToken ct)
    {
    }
}

// WorldBuilder.BuildWorldAsync 内部须遵守线程约束:
// ECS 数据生成(噪声/地形)可切线程池, EntityManager.CreateEntity 必须主线程
//
// Round 2 issue 修复: 原签名 BuildWorldAsync(CancellationToken ct) 内部使用 seed/width/height
// 既非参数也非字段,编译失败。改为接收 ArenaConfig 参数(§5.1 已声明 ArenaConfig{Width,Height,Seed,Mode,MaxSpeed},
// 详见 multiplayer-design §4.4), 内部用 config.Seed / config.Width / config.Height。
public sealed class WorldBuilder : MonoBehaviour
{
    public async UniTask BuildWorldAsync(ArenaConfig config, CancellationToken ct)
    {
        // 1. 噪声/地形等纯计算可切线程池(无 ECS 调用)
        var heightMap = await UniTask.RunOnThreadPool(
            () => NoiseCalc(config), cancellationToken: ct);

        // 2. 回到主线程后再操作 EntityManager(issue 11: CreateEntity 必须主线程)
        await UniTask.SwitchToMainThread(ct);
        var em = World.DefaultGameObjectInjectionWorld.EntityManager;  // 主线程获取
        PixelWorldFactory.BuildWorld(em, config.Width, config.Height, config.Seed, heightMap);  // 主线程 ECS 写入
        // 3. 注入 PixelGridAdapter: adapter.Initialize(sourcePixels, oldGrid) (见 §4.1.6, 与注释声明的注入来源闭环)
    }
}
```

## 6. ZLinq / ZString / UniTask 集成点

### 6.1 ZLinq 替换表格

| 集成点 | 替换前 | 替换后 | 程序集 |
|--------|--------|--------|--------|
| 查询活跃像素 | `pixels.Where(p => p.IsActive).ToList()` | `foreach (var p in pixels.AsRef().Where(p => p.IsActive))` | Simulation |
| 生物注册表查找 | `creatures.Where(c => c.Health > 0).OrderBy(c => c.Distance).ToList()` | `creatures.AsRef().Where(c => c.Health > 0).OrderBy(c => c.Distance)` | Gameplay |
| 法术目标选择 | `enemies.InCircle(center, radius).MinBy(e => e.Distance)` | ZLinq `MinBy` 零分配 | Gameplay |
| 装备背包查询 | `inventory.Where(e => e.IsEquipped).Select(e => e.Ability).ToList()` | ZLinq `AsRef().Where().Select()` | Gameplay |
| 调试统计 | `stats.Where(s => s.Mode == ChunkBased).Average(s => s.SimMs)` | ZLinq `AsRef().Where().Average()` | Editor |
| 玩家排行 | `players.OrderByDescending(p => p.Score).Take(playerCount)` | ZLinq `OrderByDescending().Take()` | Gameplay（playerCount=RoomConfig.PlayerCount） |

**关键约束**：
- **Burst Job 内禁用 ZLinq**（ZLinq 是托管类型，Job 内只能用 NativeArray 索引）
- ZLinq 仅用于主线程或 MonoBehaviour 逻辑层
- 热路径（每帧执行）的 `Where/Select/OrderBy` 必须用 `AsRef()` 零分配模式

### 6.2 ZString 替换表格

| 集成点 | 替换前 | 替换后 | 程序集 |
|--------|--------|--------|--------|
| 调试日志 | `Debug.Log($"Step {step} took {ms}ms")` | `Debug.Log(ZString.Format("Step {0} took {1}ms", step, ms))` | 全部 |
| IMGUI 覆盖层 | `GUI.Label(rect, $"FPS: {fps:F1}")` | `using var sb = ZString.CreateStringBuilder(); sb.AppendFormat(...); GUI.Label(rect, sb.ToString())` | Editor |
| 性能面板 | `Debug.Log($"Pixels: {count}, Chunks: {chunks}")` | `ZString.Format("Pixels: {0}, Chunks: {1}", count, chunks)` | Editor |
| 生物命名 | `creature.Name = $"爬行者 Lv.{level}"` | `creature.Name = ZString.Format("爬行者 Lv.{0}", level)` | Gameplay |
| 伤害数字 | `text.text = $"-{damage}"` | `ZString.Format("-{0}", damage)` | Gameplay |
| 错误上报 | `Debug.LogError($"Load failed: {path}")` | `Debug.LogError(ZString.Format("Load failed: {0}", path))` | 全部 |

**关键约束**：
- `using var sb = ZString.CreateStringBuilder()` 必须用 `using` 确保释放
- Burst Job 内禁用 ZString（Job 内不产生字符串）
- 中文文本：**禁止用 py/ps1 脚本修改含中文的 .cs 文件**（用户规则）

### 6.3 UniTask 替换表格

| 集成点 | 替换前 | 替换后 | 程序集 |
|--------|--------|--------|--------|
| 世界生成 | `IEnumerator GenerateWorld() { yield return null; }` | `async UniTaskVoid GenerateWorldAsync(CancellationToken ct)` | Simulation |
| YooAsset 资源加载 | `yield return YooAssets.LoadAssetAsync(...)` | `await handle.ToUniTask(cancellationToken: ct)` | Core/Bootstrap |
| 法术施放流程 | `IEnumerator CastSpell() { yield return ...; }` | `async UniTask CastSpellAsync(SpellElement e, CancellationToken ct)` | Gameplay |
| 装备拾取 | `IEnumerator Pickup() { yield return ...; }` | `async UniTask PickupAsync(CancellationToken ct)` | Gameplay |
| 场景切换 | `SceneManager.LoadSceneAsync(...)` 协程 | `await SceneManager.LoadSceneAsync(...).ToUniTask(ct)` | Core |
| 工作线程切换 | `Task.Run(...)` | `await UniTask.SwitchToThreadPool(); ...; await UniTask.SwitchToMainThread()` | Simulation |
| 等待帧 | `yield return new WaitForEndOfFrame()` | `await UniTask.WaitForEndOfFrame(ct)` | 全部 |

**关键约束**：
- **禁止协程**（`IEnumerator` 全部替换为 `UniTask`/`UniTaskVoid`）
- `UniTaskVoid` 用于 fire-and-forget（如 `Start()` 调用）
- `UniTask<T>` 用于需要返回值的场景
- `CancellationToken` 必须传递（避免内存泄漏）
- Burst Job 内禁用 UniTask（Job 内无异步）

## 7. DOTS 化策略

### 7.1 数据流改造（旧 → 新 → 桥接）

```
旧数据流:
  PixelGrid.Pixels[,] (托管) ←── MovementSystem (MonoBehaviour)
                            ←── InteractionSystem (MonoBehaviour)
                            ──► PixelWorldRenderer (主线程 SetPixels32)

新数据流:
  ECS Entity (PixelData) ──► MovementJob (IJobChunk, Burst)
                        ──► InteractionJob (IJobChunk, Burst)
                        ──► RenderBridgeSystem ──► PixelWorldRenderer (主线程同步)

桥接(决策 1, Phase 1-5 共存):
  PixelGridAdapter ──► 未迁移 MonoBehaviour 代码
                  ──► ECS NativeArray<PixelData> (真实数据源)
```

**PixelGridAdapter 桥接规则**：
- Phase 1 创建：MonoBehaviour 代码读 Adapter，Adapter 读 ECS
- Phase 2-5 共存：已迁移代码直接读 ECS，未迁移代码读 Adapter
- Phase 6 删除：所有 MonoBehaviour 完成迁移后移除 Adapter

### 7.2 Job 设计（5 个 Job 接口签名）

| Job | 接口 | 输入 | 输出 | 调度方式 |
|-----|------|------|------|---------|
| `MovementJob` | `IJobChunk` | `CurrentPixels`/`NextPixels`/`GridSize`/`RNG`/`FrameIndex` | mutate `PixelData` (经 NextPixels) | `ScheduleParallel(query, dep)` |
| `InteractionJob` | `IJobChunk` | `CurrentPixels`/`NextPixels`/`MaterialDefs`/`AmbientTemp`/`RNG` | mutate `PixelData` (经 NextPixels) | `ScheduleParallel(query, dep)`（依赖 Movement） |
| `PlayerMovementJob` | `IJobChunk` | `PlayerLookup`/[ReadOnly]`PixelLookup`/`GridSize`/`DeltaTime`/`ArenaConfig` | mutate `PlayerData` | `ScheduleParallel` |
| `SpellMovementJob` | `IJobChunk` | `SpellLookup`/`CurrentPixels`/`NextPixels`/`GridSize`/`DeltaTime` | mutate `SpellData` + `PixelData` (经 NextPixels) | `ScheduleParallel` |
| `CreatureAIJob` | `IJobChunk` | `CreatureLookup`/`PlayerLookup`/[ReadOnly]`PixelLookup`/`NativeQueue<PixelWriteRequest>`/`GridSize`/`DeltaTime` | mutate `CreatureData` (像素写经队列) | `ScheduleParallel` |

**统一 Burst 配置**（所有 Job）：
```csharp
[BurstCompile(
    CompileSynchronously = true,
    FloatMode = FloatMode.Strict,
    FloatPrecision = FloatPrecision.Standard)]
public struct XxxJob : IJobChunk { /* ... */ }
```

### 7.3 Burst 兼容性

| 项 | 措施 |
|----|------|
| blittable 数据 | `PixelData` 全 byte/short/ushort 字段，无 `bool`（用 `byte` 位标志替代） |
| 容器 | `NativeArray<T>` / `NativeList<T>` 替代 `List<T>` / `T[,]` |
| 字符串 | Burst Job 内**禁用**字符串（ZString 仅主线程） |
| 托管类型 | Job struct 内**禁用**任何引用类型 |
| 数学 | `Fix64` / `Unity.Mathematics.math` 替代 `System.Math` |
| RNG | `Xorshift128Plus`（struct）替代 `System.Random`（class） |
| 方法内联 | 热路径 `[MethodImpl(MethodImplOptions.AggressiveInlining)]` |
| 异常 | Burst Job 内禁用 `throw`，用返回值/错误码 |

### 7.4 多线程确定性

> 引用 [multiplayer-frame-sync-design.md §4.2.2](./多人联机帧同步对战设计.md) + [§8.1](./多人联机帧同步对战设计.md)

```
每逻辑帧 (33ms):
├── 阶段 1: 输入收集 (1ms) - 单线程
├── 阶段 2: 玩家层模拟 (5ms)
│   └── PlayerMovementJob.ScheduleParallel, chunk 排序按 Entity.Index 升序
├── 阶段 3: 环境层 CA 模拟 (15ms)
│   ├── MovementJob.ScheduleParallel(query, dep), chunk 排序按 (chunkX, chunkY) 字典序
│   ├── InteractionJob.ScheduleParallel(query, dep) (依赖 Movement)
│   └── 冲突解决: 多像素竞争同一格 → (x, y) 字典序升序先到先得
├── 阶段 4: 法术/生物 (5ms)
│   ├── SpellMovementJob.ScheduleParallel
│   └── CreatureAIJob.ScheduleParallel
├── 阶段 5: 状态快照 (2ms) - 单线程, 计算关键状态哈希
└── 阶段 6: 调和检查 (每 6 帧, 2ms) - 由 AOT 触发
```

**迭代方向**：偶数帧从左上到右下，奇数帧从右下到左上（消除方向偏置）。

## 8. 确定性改造清单

| 项 | 当前 | 目标 | 优先级 |
|----|------|------|--------|
| RNG | `System.Random` | `Xorshift128Plus`（服务器下发种子） | P0 |
| 浮点数学 | `float` / `Vector2` | `Fix64` / `Fix64Vec2` | P0 |

| 容器 | `List<T>` / `T[,]` | `NativeArray<T>` / `NativeList<T>` | P0 |
| 字符串 | `$"..."` | ZString（仅主线程） | P2 |
| 遍历顺序 | 取决于 List 顺序 | `EntityQuery` 显式排序 | P0 |

- [ ] 256×256 默认场景截图与原版像素级匹配（100 帧对比）
- [ ] 玩家移动 / 施法 / 生物 AI / 装备拾取视觉一致
- [ ] 周期调和（每 6 帧）视觉无突变（100ms 插值平滑）

### 12.4 性能验收

> 硬件基准（issue 13）：i7-12700H（Win 开发机）+ Android 中端机（骁龙 7 Gen 1 级别），两档均须满足。

- [ ] 256×256 默认场景单帧逻辑耗时 < 16ms（i7-12700H，30Hz 预算 33ms）；Android 中端机 < 25ms
- [ ] 8 核机器（i7-12700H）MovementJob 加速比 ≥ 2×（vs 单线程）；Android 中端机 ≥ 1.5×
- [ ] 热路径 GC = 0 字节/帧（Profiler GC Alloc）
- [ ] 主线程 `Texture2D.SetPixels32` 耗时 < 2ms（i7-12700H）；Android 中端机 < 5ms
- [ ] GameBootstrap 异步加载流程 < 3s（i7-12700H，256×256 默认场景）；Android 中端机 < 6s

### 12.5 代码规范验收

- [ ] 双语注释（中文 + 英文摘要）覆盖所有 public API
- [ ] `struct` 用于数据，`sealed class` 用于 MonoBehaviour
- [ ] 热路径方法标记 `[MethodImpl(MethodImplOptions.AggressiveInlining)]`
- [ ] 全局无 `IEnumerator`（UniTask 完全替代）
- [ ] Burst Job 内无字符串 / 无托管类型
- [ ] 单文件行数 < 500（Bootstrap 拆分后）
