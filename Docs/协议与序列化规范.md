## 2. 同步模型与帧率基线

| 指标 | 目标值 | 备注 |
|------|--------|------|
| tick rate | 30 Hz 逻辑 / 60 Hz 渲染 | 输入在 30Hz tick 边界结算（见输入架构 §5.1） |
| 对战规模 | 2–8 人（默认 4） | `RoomConfig.PlayerCount` |
| 输入延迟 | `INPUT_DELAY=2` 帧（66ms） | GGPO 风格（FRAME_SYNC §2.3.2） |
| 最大回滚 | `MAX_ROLLBACK=7` 帧（233ms） | 玩家层 GGPO 回滚（FRAME_SYNC §2.3） |
| 状态哈希比对 | 每 60 帧（0.5Hz） | 多数派裁决（主架构 §9.2） |

**确定性红线**：凡进入 `InputPayload` 的字段都参与锁步状态哈希（每 60 帧 `C2G_StateHash` 比对）。故 `PauseMenu` / `QuickChat` 不进 `InputPayload`（走本地 UI / `C2G_ChatMsg`），不污染哈希、不导致 desync（输入架构 §7.2）。

## 3. 双序列化分工（protobuf 低频 + MemoryPack 30Hz 高频）

项目采用 **protobuf + MemoryPack 双序列化**（`开发智能体配置.md` §3）。分工原则：

- **MemoryPack**：高频、逐帧、小包、对延迟敏感 → 比 protobuf 快 5–10×，且与 DOTS blittable 结构零分配映射。
- **protobuf**：低频、一次性、事件驱动、需要路由或可读性优先 → 用 `optional` + 保留字段号（主架构 §6.6 风格），支持 `IRouteMessage` / `IRequest` / `IResponse`。

### 3.1 消息序列化分工表

| 消息 | 方向 | 序列化 | 频率 | 分工理由 |
|------|------|--------|------|---------|
| `C2G_Input` | C→S | **MemoryPack** | 30 Hz | 逐帧热路径，极小包，需极速 |
| `G2C_FrameBatch` | S→C | **MemoryPack** | 30 Hz | 聚合所有玩家输入，逐帧热路径 |
| `G2C_SpectatorBatch` | S→C | **MemoryPack** | 30 Hz | 观战延迟帧批次，逐帧 |
| `C2G_ClockSyncRequest` / `G2C_ClockSyncResponse` | C↔S | **MemoryPack** | 每 10s | 小包高频，需极速 |
| `C2G_StateHash` | C→S | **MemoryPack** | 0.5 Hz | 小包，确定性强，走热路径通道 |
| `G2C_SnapshotChunk` | S→C | **MemoryPack** | 重连分片 | 16KB 大块，MemoryPack 序列化快照快 |
| `G2C_SceneEvent` | S→C | **protobuf + Route** | 事件驱动 | 定向投递到房间实体，帧延迟加密（主架构 §6.5） |
| `C2G_ChatMsg` / `G2C_ChatMsg` | C↔S | protobuf | 用户驱动 | 文本/短语，低频 |
| `C2G_MatchRequest` / `G2C_MatchResult` | C↔S | protobuf | 一次性 | 匹配请求/结果 |

| `C2G_DeviceReport` | C→S | protobuf | 每次连接（一次性） | 稳定匿名设备标识上报（DeviceId/InstallId，哈希后）；dataanalyst 埋点约束 #4 前置；不进 30Hz 输入流，不 bump `ProtocolVersion` |
| `C2G_ReconnectRequest` / `G2C_ReconnectResponse` / `C2G_ReconnectAbort` / `G2C_ReconnectProgress` | C↔S | protobuf | 一次性/重连期 | 重连流程，低频 |

| `G2C_MatchSettle` | S→C | protobuf | 一次性（对局结束） | 赛后结算下行，承载 `ResultPayload`（名次/统计/MMR/回放可用性）；不进 30Hz 输入流，不 bump `ProtocolVersion`（见 §6.1） |

### 3.2 序列化实现约束

- **schema 单一源**：所有消息（含 MemoryPack 标注的）在 `Tools/NetworkProtocol/Outer/OuterMessage.proto` 中**统一定义**，每条消息上方用 `// Protocol MemoryPack` 或 `// Protocol ProtoBuf` 标注（沿用 FRAME_SYNC §7.2 风格）。
- **MemoryPack 热路径**（`C2G_Input` / `G2C_FrameBatch` / `G2C_SpectatorBatch` / `C2G_StateHash` / 时钟同步 / `G2C_SnapshotChunk`）运行时**仅走 MemoryPack 序列化**，不使用 protobuf 编码；protobuf 定义仅作为 schema 文档与代码生成基准。
- **禁破坏性 schema 变更**（MemoryPack）：字段顺序/类型/编号一旦发布不可改；新增字段只能追加，删除字段必须 bump `ProtocolVersion`（§7）。
- **Hotfix 边界**：协议处理 handler 在 `Hotfix.asmdef`；模拟逻辑（`NoitaCA.Lockstep` / `NoitaCA.Simulation`）在 AOT，**绝不在 Hotfix**（主架构 §4.1）。

## 4. InputPayload 完整定稿（8 字节）

> **本表为最终定稿**。相对主架构 §6.2：删除 `SpellId`（见 §4.3 与 A4 结论）。相对输入架构 §2.3：移除"SpellId 恒=0"表述。

### 4.1 字段表

| 字段 | 类型 | 位宽 | 范围 / 编码 | 量化? | 边沿/持续 | 状态 |
|------|------|------|------------|-------|----------|------|
| `MoveX` | `int16` | 16 | `[-100, 100]`，归一化向量 ×100 | 是（×100） | 持续 | 使用 |
| `MoveY` | `int16` | 16 | `[-100, 100]`，归一化向量 ×100 | 是（×100） | 持续 | 使用 |
| `ActionFlags` | `uint16` | 16 | 位标志（见 §5 位分配表） | 否（位） | 边沿 | 使用 |
| `AimAngle` | `int16` | 16 | `[0, 628]`，×100 表示 `[0, 2π)` | 是（×100） | 持续 | 使用 |

**wire 体积**：`2+2+2+2 = 8 字节`（原含 `SpellId` 为 12 字节，**−33%**）。

### 4.2 量化细则（确定性核心）

#### MoveX / MoveY（归一化向量）
- 采集层产出归一化向量（键盘 8 向 / 触屏摇杆偏移），经 `Core.NormalizeMove()` 统一处理：
  - `len = Fix64Math.Sqrt(x²+y²)`；若 `len > 100`（含浮点误差容忍）则 `(x,y) = (x,y)*100/len`。
  - `MoveX = clamp(round(x*100), -100, 100)`，`MoveY` 同（屏幕 Y 轴翻转 → 世界 Y 取负）。
- 模长严格 ≤ 100。服务器 `AntiCheatSystem.ValidateInput`（主架构 §9.2）校验 `|MoveX|≤100 ∧ |MoveY|≤100 ∧ √(MoveX²+MoveY²)≤100`。
- **PC 与移动端必须共用同一 `NormalizeMove()`**（输入架构 §3.4 / §4.4 / §7.3）。

#### AimAngle（瞄准角）
- `rawAngle = Fix64Math.Atan2(v.y, v.x)`（确定性定点，**禁止** `Math.Atan2` 浮点发散）。
- `AimAngle = clamp(round(rawAngle / (2π) * 628), 0, 628)`。
- 分辨率 ≈ 0.01 rad（≈0.57°），满足像素级命中（输入架构 §3.2）。
- 仅量化后整数进流；原始鼠标坐标 / 光标位置不进模拟（回滚以权威 `InputPayload` 覆盖）。

### 4.3 已删除字段：SpellId（A4 结论）

**结论：从 `InputPayload` 删除 `uint32 SpellId`。**

理由：
1. **孤儿字段**：循环切换模型下，实际施放法术 = `PlayerData.SpellIds[PlayerData.SelectedSpellSlot]`。`SelectedSpellSlot` 由 `ActionFlags.SelectNext/SelectPrev` 确定性自增/自减（0–3 循环），是 `PlayerData` 字段（进快照、可回滚重放）。模拟 `PlayerMovementJob.CastSpell(player, player.SpellIds, player.SelectedSpellSlot)` **从不读取** per-frame `SpellId` → 该字段在确定性流中无任何消费者。
2. **带宽浪费**：每帧白占 4 字节，占原 12 字节 payload 的 1/3；与 §13 风险登记"N=8 带宽过高"目标冲突。
3. **确定性隐患**：恒=0 的字段若被客户端误置非零值，会污染状态哈希且当前 `ValidateInput` 未校验 → 成为 desync 隐患。
4. **删除安全**：所有上游文档均为"待评审"状态（未发布），无已上线客户端需迁移；旧客户端由 §7 版本校验拒绝进对局。

**迁移动作**：删除触发 `ProtocolVersion` 1 → 2 bump（§7）。未来若需"直选某槽法术"类语义，按 §11.3 回溯机制新增**动作位/新字段**，不复用已删除字段。

## 5. InputActionFlags 位分配表（最终定稿，16 bit uint16）

> **本表为最终定稿，取代主架构 §6.2 注释与输入架构 §2.2.1**。A5 结论：bit2/3 不再作为"直选槽"语义使用，降级为**保留位（Reserved），当前模型下禁止置位**。

| Bit | 枚举名 | 状态 | 语义 | 来源 |
|-----|--------|------|------|------|
| 0 | `Jump` | **使用** | 跳跃 | 既有 |
| 1 | `Attack` (=`CastSpell`) | **使用** | 释放当前槽法术（`SpellIds[SelectedSpellSlot]`） | 既有 |
| 2 | `Spell1` | **保留位 (Reserved)** | 直选槽1；循环切换模型下**禁止置位**，服务器拒绝 | 既有 / 兼容保留 |
| 3 | `Spell2` | **保留位 (Reserved)** | 直选槽2；循环切换模型下**禁止置位**，服务器拒绝 | 既有 / 兼容保留 |
| 4 | `Pickup` | **使用** | 拾取 | 既有 |
| 5 | `Drop` | **使用** | 丢弃 | 既有 |
| 6 | `SelectNext` | **使用** | 切换法术（下一槽，0–3 循环） | 新增 |
| 7 | `SelectPrev` | **使用** | 切换法术（上一槽，0–3 循环） | 新增 |
| 8 | `UseConsumable` | **使用** | 使用消耗品 | 新增 |
| 9 | `Dash` | **使用** | 冲刺/闪避（待 PM 确认，见 §6） | 新增 |
| 10–15 | （未分配） | **保留位 (Reserved)** | 未来扩展动作；当前强制 0 | 预留 |

### 5.1 A5 结论：Spell1/Spell2(bit2/3) 处置

**结论：保留为保留位（Reserved），协议层明确禁止置位；服务器 `ValidateInput` 拒绝任何非零的 bit2/3。不立即复用为其他动作。**

推理：
- **互斥性**："循环切换"（SelectNext/Prev 增量维护 `SelectedSpellSlot`）+ 单施法（`Attack`）与"直选槽 Spell1/Spell2"是两套**互斥**的选择范式。Q6 已定"循环切换 + 单施法"为唯一上线模型 → 直选语义在当前模型下是死代码。
- **保留而非复用**：`uint16` 仍有 bit10–15 共 6 个空闲位供未来扩展动作（输入架构 §11.3 已约定新动作走 bit10–15），**无需**占用 bit2/3。保留其名（`Spell1`/`Spell2`）以兑现"兼容保留通道"承诺——若未来引入混合直选模式，可无 schema 变更地复用这两位（仍受 §7 版本约束）。
- **明确"留废"**：状态 = "保留位 / 当前模型禁止置位"。既非"使用"亦非"已删除"，而是**显式禁用**——消除协议层悬而未决。

### 5.2 互斥与回滚约束

- `SelectNext` / `SelectPrev` 与 `Spell1` / `Spell2` 在任一帧**不得同时置位**；若未来启用直选，`ValidateInput` 应拒绝同时含两类位的帧。
- `SelectNext` / `SelectPrev` / `UseConsumable` / `Dash` 与 `Jump` / `Attack` **同等对待**：全部进 `DeterministicInput`、全部入 `InputRingBuffer`、回滚 `ReplayForward` 时全部重放（输入架构 §6.3）。`SelectedSpellSlot` 由 SelectNext/Prev 位推导，回滚重放即重建，无需把槽号写入 `InputPayload`。
- 边沿语义：长按不连发（冷却由服务器校验）；`EdgeBuffer` 在 30Hz tick 边界结算"本 tick 内是否按下"（输入架构 §3.3）。

### 5.3 枚举定义（AOT, blittable, uint16）

```csharp
// NoitaCA.Core/Input/InputAction.cs
// 位分配以 协议与序列化规范.md §5 为准（收口单一事实源）
[Flags]
public enum InputActionFlags : ushort
{
    None          = 0,
    Jump          = 1 << 0,  // 使用: 跳跃
    Attack        = 1 << 1,  // 使用: 释放当前槽法术 (=CastSpell)
    Spell1        = 1 << 2,  // 保留位: 直选槽1，当前模型禁止置位（服务器拒绝）
    Spell2        = 1 << 3,  // 保留位: 直选槽2，当前模型禁止置位（服务器拒绝）
    Pickup        = 1 << 4,  // 使用: 拾取
    Drop          = 1 << 5,  // 使用: 丢弃
    SelectNext    = 1 << 6,  // 使用: 切换法术(下一槽)
    SelectPrev    = 1 << 7,  // 使用: 切换法术(上一槽)
    UseConsumable = 1 << 8,  // 使用: 使用消耗品
    Dash          = 1 << 9,  // 使用: 冲刺/闪避（PM 确认后生效）
    // bit10-15: 保留位，当前强制 0
}
```

## 6. 包装消息定稿（MemoryPack 热路径）

```protobuf
// Protocol MemoryPack
// 高频 30Hz，逐帧确定性输入；C→S
message C2G_Input // IMessage
{
    uint64 Frame = 1;          // 客户端帧号
    InputPayload Payload = 2;  // 输入载荷（见 §4，8 字节）
}

// Protocol MemoryPack
// 子消息必须显式标注：ProtocolExportTool（2026.0.1023）不会从父消息推导序列化类型，
// 不标注会默认生成 ProtoBuf，导致 MemoryPack 嵌套序列化失败（已实测）。
message InputPayload
{
    int16  MoveX = 1;           // [-100, 100] 归一化 ×100
    int16  MoveY = 2;           // [-100, 100] 归一化 ×100
    uint16 ActionFlags = 3;     // 位标志（§5：bit2/3 保留禁用）
    int16  AimAngle = 4;        // [0, 628] ×100 表示 [0, 2π)
}

// Protocol MemoryPack
message G2C_FrameBatch // IMessage
{
    uint64 Frame = 1;                    // 服务器帧号
    repeated PlayerInput Inputs = 2;     // 所有玩家输入
}

// Protocol MemoryPack
message PlayerInput
{
    uint32 PlayerId = 1;
    InputPayload Payload = 2;
}

// Protocol MemoryPack
// 观战延迟帧批次；S→C
message G2C_SpectatorBatch // IMessage
{
    uint64 Frame = 1;
    repeated PlayerInput Inputs = 2;
    // repeated G2C_SceneEvent DelayedEvents = 3;  // 该帧触发的场景事件
    //   ↑ G2C_SceneEvent 尚未定义（见 §3.1，Route 消息），其完整 schema + RouteType.Config
    //     注册属观战模式（后 MVP）；当前 OuterMessage.proto 已注释此字段以免导出期未定义类型。
    //     启用观战时再补（field 3）。
    uint32 SpectatorId = 4;
}
```

> **注意**：`SpellId` 已从 `InputPayload` 移除；`PlayerData.SpellIds`（玩家法术槽位 loadout，4 槽）与 `PlayerData.SelectedSpellSlot`（当前槽，0–3）为玩家状态字段，由快照复制，**不**经 `InputPayload` 传输（输入架构 §2.5）。

### 6.1 赛后结算下行（ProtoBuf 低频）

> **定位**：对局状态机进入 `MatchEnded` 后，服务端下发的权威结算包（承载 tucai `结算界面规格 v1.1 §3.3` 的 `ResultPayload`/`PlayerResult`）。**低频一次性、走 protobuf、不进入 30Hz `InputPayload`、不触发 `ProtocolVersion` bump**（与 `C2G_MatchResult` 同档）。
>
> **与 tucai §3.3 字段的对应关系**：本消息是 wire 形态。注意——`Rank` / `Kills` / `Deaths` / `MaterialContribution` 等均为**确定性模拟量**，客户端在 `MatchEnded` 时已可从本地 sim 状态得出；本消息主要提供服务端权威的 **`MmrDelta`（非确定性，ELO 变体 K=32）** 与跨端一致性校验基准。`DisplayName` 由客户端按 `PlayerId` 从房间名册查得；`IsWinner`/`IsDraw` 由 `Rank` 与 `Players[]` 推导（多人同 `Rank=1` 即平局）；`MatchDurationMs` 由 `TotalFrames / 30 * 1000` 推导。若 team-lead / tucai 要求服务端全量下发，再扩展 `SettlePlayer`（前向兼容追加字段，不破坏现有 wire）。

```protobuf
// Protocol ProtoBuf
// 低频一次性，赛后结算下行；不进 30Hz 输入流，不 bump ProtocolVersion（与 C2G_MatchResult 同档）
// 承载 tucai 结算界面规格 v1.1 §3.3 的 ResultPayload/PlayerResult（最终字段以本规范为准）
message G2C_MatchSettle // IMessage
{
    uint32 MatchId = 1;             // 对局 ID
    uint32 TotalFrames = 2;         // 本局总逻辑帧
    repeated SettlePlayer Players = 3;  // 按 Rank 升序（1 = 冠军在前）
    uint32 YourselfRank = 4;        // 本地玩家名次（客户端已知 PlayerId，可派生；服务端给更省事）
}

// 随父消息 G2C_MatchSettle(ProtoBuf) 推导，无需单独标注 // Protocol
message SettlePlayer
{
    uint32 PlayerId = 1;
    uint32 Rank = 2;                // 名次（1 = 冠军）
    uint32 Kills = 3;               // K
    uint32 Deaths = 4;              // D（0 或 1）
    uint32 MaterialContribution = 5;    // 材料改造贡献（= tucai §3.3 MaterialModifiedPixels）
    int32  MmrDelta = 6;            // MMR 变化（排位，ELO 变体 K=32；非确定性，服务端算）
}
```

> **回放可用性**：`ReplayAvailable` 当前恒 `false`（MVP 隐藏「查看回放」按钮，见 tucai §5.4/§9.2），客户端可硬编码 `false`；若 P4+ 启用回放，作为 `G2C_MatchSettle` 的前向兼容追加字段，不破坏现有 wire。

## 7. 协议版本与兼容性

| 项 | 值 | 说明 |
|----|----|------|
| 当前 `ProtocolVersion` | **2** | 相对基线 1，因删除 `SpellId`（§4.3）bump |

**强制约束**：
- 删除 `SpellId` 属破坏性 schema 变更 → 必须 bump `ProtocolVersion` 且更新客户端版本白名单校验。
- bit2/3 保留位禁用、bit10–15 保留位禁用：**不**触发版本 bump（向后兼容），但 `ValidateInput` 必须拒绝置位。
- 任何未来新增动作/字段，遵循输入架构 §11.3 回溯机制：新增位走 bit10–15；新增字段走追加字段 + bump `ProtocolVersion`。

## 8. 序列化约束与反作弊

服务器 `AntiCheatSystem.ValidateInput`（主架构 §9.2）对 `InputPayload` 的校验**扩展**为：

```csharp
public bool ValidateInput(uint playerId, InputPayload input, long frame)
{
    // 1. 帧频率：dt = frame - LastInputFrame < 1 → 拒绝（同帧/乱序）
    if (frame - player.LastInputFrame < 1) return false;

    // 2. 移动向量：范围 + 模长
    if (Math.Abs(input.MoveX) > 100 || Math.Abs(input.MoveY) > 100) return false;
    if (Math.Sqrt((long)input.MoveX*input.MoveX + (long)input.MoveY*input.MoveY) > 100) return false;

    // 3. 瞄准角范围
    if (input.AimAngle < 0 || input.AimAngle > 628) return false;

    // 4. ActionFlags 保留位禁用（A5）：bit2/3 与 bit10-15 必须全 0
    const ushort FORBIDDEN = (1<<2) | (1<<3) | 0xFC00; // bit2,3 + bit10-15
    if ((input.ActionFlags & FORBIDDEN) != 0) return false;

    // 5. 位移幅度（防瞬移，基于 PlayerData.Position，非 input 本身）
    //    maxMove = ArenaConfig.MaxSpeed / 30 * 1.5
    //    ...（主架构 §9.2 原文）

    return true;
}
```

**客户端侧**：`NoitaCA.Lockstep` 序列化适配在 `Capture()` 产出 `DeterministicInput` 后，确保保留位恒为 0（采集层不置 bit2/3、bit10–15）；MemoryPack 序列化 `InputPayload` 发 `C2G_Input`。
