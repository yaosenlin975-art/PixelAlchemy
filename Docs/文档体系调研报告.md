## 1. 背景与目标

NoitaCA 当前 `Docs/` 目录下已有 4 份文档，全部为**技术架构 / 方案类**：

| 现有文档 | 学科归类 | 覆盖内容 |
|---------|---------|---------|
| `多人联机帧同步对战设计.md` | 技术架构 / TDD | 多人帧同步总体架构、同步模型、四大系统、反作弊、实施路线 |
| `帧同步Netcode设计.md` | 技术架构 / TDD | 分层回滚、输入预测、断线重连、时钟同步、协议补充 |
| `玩法重构方案.md` | 技术架构 / TDD | DOTS ECS 玩法重构方案、程序集、Job 接口、6 阶段路线 |
| `开发智能体配置.md` | 技术配置 / 约定 | 项目配置、技术栈、编码规范、架构边界、陷阱 |

**结论**：技术架构类已被充分覆盖，但项目**完全缺失**游戏设计、项目搭建、美术风格、测试 QA、构建部署类文档。本报告完成两件事：

1. 整理「游戏开发标准文档体系」（第 2 步成果）
2. 对照本项目做缺口分析，给出补齐清单与暂缓理由（第 3 步成果）

## 2. 游戏开发标准文档体系（业界参考）

按学科分类。每项标注：**作用 / 适用阶段 / 产出时机**。

### 2.1 设计类（Design）

| 文档 | 作用 | 适用阶段 | 产出时机 |
|------|------|---------|---------|
| **GDD 游戏设计文档** | 定义核心循环、胜负规则、玩法系统、体验目标，是全团队共识的"游戏是什么" | 立项 → 全程 | 概念期产出 v1，随开发迭代 |
| 关卡 / 世界设计 | 关卡布局、竞技场、难度曲线、引导 | 原型后 | 核心循环稳定后 |
| 数值 / 平衡设计 | 伤害、冷却、经济、掉落权重、MMR | 原型期起 | 玩法可玩后即开始 |
| 叙事 / 剧情设计 | 世界观、角色背景、过场 | 视项目 | 内容型项目早期 |

### 2.2 美术类（Art）

| 文档 | 作用 | 适用阶段 | 产出时机 |
|------|------|---------|---------|
| **美术风格指南（Art Bible）** | 像素网格、调色板、材质/角色/UI 视觉规范，保证视觉一致 | 原型前 → 全程 | 美术动工前定基线 |
| 动画 / 特效规范 | 帧率、缓动、粒子、Telegraph 风格 | 美术期 | 与美术指南同期 |
| UI 视觉规范 | 字体、间距、图标、状态色 | UI 动工前 | 与美术指南同期 |

### 2.3 音频类（Audio）

| 文档 | 作用 | 适用阶段 | 产出时机 |
|------|------|---------|---------|
| 音频设计文档 | 音效清单、BGM、空间化、音量矩阵 | 内容填充期 | 有可玩内容后 |

### 2.4 技术 / 架构类（Tech）

| 文档 | 作用 | 适用阶段 | 产出时机 |
|------|------|---------|---------|
| 技术设计文档 TDD | 模块设计、接口、数据结构 | 架构期 | 编码前 |
| 架构总览 | 分层、依赖、部署拓扑 | 架构期 | 早期 |
| API / 协议文档 | 接口契约、协议字段 | 联调前 | 协议定稿后自动生成 |

### 2.5 生产 / 项目类（Production）

| 文档 | 作用 | 适用阶段 | 产出时机 |
|------|------|---------|---------|

## 3. 缺口分析（对照本项目）

> 判定基准：本项目处于**技术原型 → 首个可玩版本**过渡期，单人全职开发，已有 4 份技术架构文档，尚无任何设计 / 美术 / 搭建 / 测试 / 构建类独立文档。

| 文档类别 | 业界标准是否需要 | 本项目是否已有 | 本项目当前是否需要 | 优先级 | 理由 |
|---------|:---:|:---:|:---:|:---:|------|
| GDD 游戏设计文档 | ✅ | ❌ | ✅ **需要补齐** | P0 | 现有 4 份全是技术方案，"游戏怎么好玩"从未成文；数值/胜负/材料体系散落在代码注释，需收敛为设计共识 |
| 搭建 / Onboarding 贡献指南 | ✅ | ⚠️ 部分（开发智能体配置 有约定，但非可操作搭建步骤） | ✅ **需要补齐** | P0 | 路径已扁平化、ExporterSettings 已修、NuGet 待锁；新协作者无法仅凭 Dev-Agent 跑起来，需可操作 SOP |
| 美术风格指南 Art Bible | ✅ | ❌ | ✅ **需要补齐** | P0 | 像素落沙强依赖确定性渲染，`PixelData.Color` 为 RGB565（16 位），颜色必须可编码；需锁定调色板与材质视觉，否则热更 desync |
| 测试 / QA 计划 | ✅ | ❌ | ✅ **需要补齐** | P0 | 确定性回归、三平台哈希一致、Burst 锁定等验收诉求已写在架构文档里但无独立 QA 计划与 CI 集成方案 |
| 构建部署 / CI 文档 | ✅ | ❌（仅架构文档内联提及） | ✅ **需要补齐** | P1 | HybridCLR 热更 dll 签名、YooAsset 清单、Burst 版本锁、协议灰度已在架构文档定义，但缺独立构建 / 发布 / 回滚流程文档 |
| 技术架构 / TDD | ✅ | ✅ 已有 3 份 | ➖ 已覆盖 | — | 帧同步、netcode、DOTS 重构均已成文，无需重复 |
| 协议 / API 文档 | ✅ | ⚠️ 协议定义在 `.proto` + 架构文档 §6/§7 | ➖ 基本覆盖 | — | 协议以 `.proto` 为权威源，架构文档已整理消息表，后续可由导出工具自动生成 |
| 风险评估 | ✅ | ⚠️ 架构文档 §13/§10 含风险登记 | ➖ 基本覆盖 | — | 已有风险登记，可并入路线图维护 |
| 路线图 / 里程碑 | ✅ | ⚠️ 架构文档 §11 实施路线 | ➖ 基本覆盖 | — | 7 阶段路线图已存在，可视为里程碑 |
| 数值 / 平衡设计 | ✅ | ⚠️ 道具权重 / MMR 散落架构文档 | ⏸ 暂缓（低优先） | P2 | 基础数值已定（道具刷新权重、ELO），深度平衡需可玩后据 QA 数据迭代 |
| 关卡 / 世界设计 | ✅ | ❌ | ⏸ 暂缓 | P2 | 竞技场为服务器种子确定性生成 + 可选边界模式，无手设计关卡需求；待玩法验证后再做关卡池 |
| 音频设计文档 | ✅ | ❌ | ⏸ **暂缓** | — | 项目阶段过早，尚无音效资源与音频工程师；音频不影响逻辑确定性 |
| 可玩性测试 | ✅ | ❌ | ⏸ **暂缓** | — | 依赖可玩构建，须先完成 GDD + QA 计划；当前无内容可测 |
| 本地化 | ✅ | ❌ | ⏸ **暂缓** | — | 当前纯中文项目，无出海计划；早期做本地化收益低 |
| 商业化 / 直播 | ✅ | ❌ | ⏸ **暂缓** | — | 无付费设计、无观战商业化诉求（观战延迟机制已做但非变现）；上线前再议 |
| 数据分析 | ✅ | ❌ | ⏸ **暂缓** | — | 无线上运营、无埋点；公测后启动 |
| 叙事 / 剧情设计 | ✅ | ❌ | ⏸ **暂缓** | — | 大逃杀 / 乱斗品类弱叙事，世界观可后续轻量补充 |

### 3.1 关键结论

- **已有（无需新建）**：技术架构 / TDD、协议 / API、风险评估、路线图 —— 被现有 4 份文档覆盖。
- **当前需补齐（本次新建 6 份）**：
  1. `文档体系调研报告.md`（本报告）
  2. `游戏设计文档.md`
  3. `项目搭建与贡献指南.md`
  4. `美术风格指南.md`
  5. `测试QA计划.md`
  6. `构建与部署.md`
- **暂缓项（理由见上表）**：音频设计、可玩性测试、本地化、商业化 / 直播、数据分析、叙事设计、数值 / 平衡深化、关卡 / 世界设计。

### 3.2 暂缓项说明（一句话）

- **音频 / 可玩性测试 / 数据分析 / 本地化 / 商业化 / 叙事**：项目处于技术原型期，无内容、无运营、无协作者专职，投入产出比低，待首个可玩版本（Phase 6 后）再启动。
- **数值深化 / 关卡设计**：基础数值已嵌入架构文档（道具权重、MMR），深度平衡与关卡池必须在 GDD + QA 数据基础上迭代，提前写易与代码脱节。
